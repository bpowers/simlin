# LTM Cross-Element Aggregate Scoring — Phase 5: Reducer hoist (aggregate nodes)

**Goal:** Treat each maximal inlined array-reducer subexpression (`SUM(pop[*])`, `MEAN(...)`, `SUM(m[D1,*])`, ...) as an implicit synthetic auxiliary node — the "aggregate node", named `$⁚ltm⁚agg⁚{n}` — so causality routes `source[d] → agg → target` instead of all-pairs `source[d] → target[e]`, both halves get real per-element link scores that compose by the chain rule, and the agg node is *trimmed* from a loop when the loop is reported (mirroring how the LTM papers trim a macro's hidden internal nodes). With reducers handled as aggregate nodes, the diagonal conflation (`share[r] = pop[r]/SUM(pop[*])`: is `pop[r]→share[r]` the numerator effect, the SUM effect, or both?) is resolved by construction — the numerator path and the SUM path become distinct loops, scored separately.

**Architecture:** A new salsa-friendly helper `enumerate_agg_nodes(model) -> Map<canonical_reducer_subexpr, agg_ident>` (deterministic; canonicalization via printed-equation-text equality since `Expr2` is not `Eq`; identical subexpressions dedupe; a variable whose entire dt-equation is exactly one reducer call maps to *itself* — no synthetic minted) is consumed by both `model_element_causal_edges` (which reroutes a Wildcard/DynamicIndex reducer reference through the agg node: `pop[d] → agg` + `agg → target[e]` instead of all-pairs) and `model_ltm_variables` (which emits the `$⁚ltm⁚agg⁚{n}` auxiliaries — ordinary computed auxes so `PREVIOUS(agg)` is available — plus the two link-score families: `pop[d] → agg` via the Phase-4 reducer machinery, and `agg → target` as a Bare scalar→arrayed / A2A link score; the LTM partial-equation builder textually substitutes a recognized reducer subexpression with its agg name). **Model equations are NOT rewritten** — the simulation evaluates the inline reducer; the agg aux yields the same value. Loop *reporting* trims agg nodes (collapse `pop[d] → agg → target[e]` to `pop[d] → target[e]`); the loop-score *equation* is built on the un-trimmed path.

**Tech Stack:** Rust; `simlin-engine`; salsa tracked functions (`model_element_causal_edges`, `model_ltm_variables` — and a new tracked `enumerate_agg_nodes`); the `Expr2` AST + `BuiltinFn` reducers; `compute_layout` / `assemble_module` LTM wiring (`compile_ltm_equation_fragment`, `build_stub_variable`); the `ltm` module's `Loop`/`Link` types; the Phase-4 arrayed-result reducer machinery.

**Scope:** Phase 5 of 6 from `docs/design-plans/2026-05-09-ltm-503-cross-element-agg.md`. **This is the conceptual core and the largest phase.** It is structured below as three subcomponents (A: enumerate + graph reroute; B: agg as a computed variable + link-score halves + the runlist-ordering fix; C: loop-reporting trim). Each subcomponent ends with passing tests. **Risk flag (design R4 + the codebase-investigator's recommendation):** if any subcomponent's diff balloons well beyond what the tasks below describe — especially the runlist-ordering fix (D4 below) or the loop-trimming pass (D1 below, which has *no* existing precedent despite the design's claim) — STOP and surface to the user; this phase is a candidate for being split into separate plan rounds.

**Codebase verified:** 2026-05-09 (codebase-investigator).

## Acceptance Criteria Coverage

This phase implements and tests:

### ltm-503-cross-element-agg.AC4: Inlined reducers become aggregate nodes
- **ltm-503-cross-element-agg.AC4.1 Success:** For `share[r] = pop[r] / SUM(pop[*])` with `share` feeding back into `pop`, `model_ltm_variables` emits a synthetic aux `$⁚ltm⁚agg⁚0` with equation `SUM(pop[*])`; the element graph contains `pop[d] → $⁚ltm⁚agg⁚0` for every element `d` and `$⁚ltm⁚agg⁚0 → share[r]` for every element `r`, and contains no direct `pop[d] → share[e]` Wildcard-derived edges.
- **ltm-503-cross-element-agg.AC4.2 Success:** A cross-element feedback loop through the aggregate is enumerated traversing `$⁚ltm⁚agg⁚0` twice; the reported `Loop` has the agg node trimmed (its links are `pop[d] → share[e]` style); its `loop_score` equation is the product of the actual per-element link scores along the un-trimmed path, including the `pop[d]→agg` and `agg→share[e]` halves.
- **ltm-503-cross-element-agg.AC4.3 Success:** A variable whose entire dt-equation is exactly one reducer call (`total_population = SUM(population[*])`) mints no synthetic agg node; `total_population` itself is the aggregate node, and behavior is otherwise as in AC4.1/4.2.
- **ltm-503-cross-element-agg.AC4.4 Success:** Nested reducers (`x = SUM(a[*]) / SUM(b[*])`) mint two distinct agg nodes; two textually-distinct-but-AST-identical reducer subexpressions within a model dedupe to one agg node.
- **ltm-503-cross-element-agg.AC4.5 Success (heterogeneous magnitudes -- the issue's motivating case):** For a 2-region `share[r] = pop[r] / SUM(pop[*])` model with `pop[big] >> pop[small]`, the relative loop scores of the cross-element loops differ measurably from what a single lumped `|Δ_aggregate(share[r]) / Δshare[r]|` link score would give, and match a hand calculation -- the per-element `|Δpop[d] / Δagg|` factor is present and non-constant across `d`.
- **ltm-503-cross-element-agg.AC4.7 Success:** Discovery mode on the AC4.1 (inlined-reducer) model routes loop-finding *through* the aggregate node — the rerouted element graph makes the `pop[d] → agg → share[e]` edges the ones discovery's strongest-path DFS traverses (rather than the pre-Phase-5 over-approximated `pop[d] → share[e]` Wildcard cross-product), and the discovered loop's score is the product of the un-trimmed per-element link scores including the `pop[d]→agg` and `agg→share[e]` halves; the aggregate node is then trimmed from the *reported* `FoundLoop.links` (AC4.2). Note: strongest-path discovery reports one loop per stock node, so a genuinely cross-element loop (`pop[big] → agg → share[small] → … → pop[small] → agg → share[big] → … → pop[big]`) is generally *not* the reported strongest loop for the `share[r] = pop[r]/SUM(pop[*])` fixture — the self-element-through-agg loop is. AC4.7's intent is "discovery is rerouted through the agg node and scores the loop on the un-trimmed path", not "the cross-element loop is the reported strongest loop". (Verified by `tests/simulate_ltm.rs::test_discovery_loop_through_agg_scored_on_untrimmed_path`.)

> Phase 5 depends on Phase 4 (arrayed-result reducer support — needed when a hoisted reducer is a partial reduce inside an arrayed target) and Phases 1-3. AC4.6 (the partial-reduce machinery) was Phase 4's deliverable; Phase 5 exercises it through synthetic agg nodes where the loop dynamics warrant.

## Context for the implementer (read before starting)

### The idea, concretely

`share[r] = pop[r] / SUM(pop[*])`. Today the element graph emits `pop[d] → share[e]` for *every* `(d, e)` (the conservative Wildcard cross-product), and `model_ltm_variables` emits a per-shape "Wildcard" link score `$⁚ltm⁚link_score⁚pop→share⁚wildcard` whose magnitude is `|Δ_aggregate(share[r]) / Δshare[r]|` — "how much of `Δshare[r]` came from the whole `SUM` moving", which omits each `pop[d]`'s *fractional* contribution to the aggregate's velocity (exactly the factor that matters when elements have very different magnitudes). After Phase 5: a synthetic aux `$⁚ltm⁚agg⁚0 = SUM(pop[*])` exists (computed during simulation, so `PREVIOUS($⁚ltm⁚agg⁚0)` works); the element graph has `pop[d] → $⁚ltm⁚agg⁚0` (one per source element) and `$⁚ltm⁚agg⁚0 → share[r]` (one per target element); the link scores are `$⁚ltm⁚link_score⁚pop[d]→$⁚ltm⁚agg⁚0` (the per-element reducer link score, from Phase 4's machinery — = `|Δpop[d] / Δagg|` essentially) and `$⁚ltm⁚link_score⁚$⁚ltm⁚agg⁚0→share` (a Bare scalar→arrayed link score — the partial of `share[r]`'s equation w.r.t. `agg` live, with `pop[r]` and everything else at PREVIOUS — i.e. the existing "Wildcard" semantics, now correctly attributed to the aggregate). The loop `pop[nyc] → agg → share[boston] → ... → pop[boston] → agg → share[nyc] → ... → pop[nyc]` (with `agg` appearing twice) is enumerated; its loop-score equation is the honest product of the link scores along it; the *reported* `Loop` has `agg` trimmed (`pop[nyc] → share[boston]`, `pop[boston] → share[nyc]`). The Bare numerator `pop[r]` still produces its own `$⁚ltm⁚link_score⁚pop→share` Bare link score and its own (non-cross-element) loop — distinct from the SUM path.
