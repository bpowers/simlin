# Test Requirements: LTM Cross-Element Aggregate Scoring (GH #503)

This document maps every acceptance criterion from
[`docs/design-plans/2026-05-09-ltm-503-cross-element-agg.md`](../../design-plans/2026-05-09-ltm-503-cross-element-agg.md)
(`ltm-503-cross-element-agg.AC1.1` ... `AC7.2`, 22 criteria) to either an
automated test (criterion text, test type, expected file path, what it verifies)
or a documented human-verification procedure (justification + approach).

It is rationalized against the implementation decisions made during planning
(phases 1 through 6 of the now-removed implementation plan) -- in particular:
Phase 1 changes `LtmSyntheticVar.equation` from `String` to
`datamodel::Equation` (tests inspecting `.equation` must destructure the
variant); Phase 2 keeps element subscripts on cross-element `Loop` links and
emits `"$⁚ltm⁚link_score⁚{from}→{to}"[elem]` references; Phase 3 names
scalar-source → arrayed-target link scores `$⁚ltm⁚link_score⁚{from}→{to}[{elem}]`
(one scalar var per target element, `dimensions: vec![]`); Phase 4 adds
arrayed-result reducer support (`$⁚ltm⁚link_score⁚matrix[d1,d2]→agg[d1]` per
`(d1,d2)` pair); Phase 5 introduces the `$⁚ltm⁚agg⁚{n}` synthetic family,
reroutes the element graph through agg nodes, and trims agg nodes when reporting
loops; Phase 6 retires the `⁚wildcard`/`⁚dynamic` per-shape link-score path.

## How the `test-analyst` agent uses this file

During execution, after each phase's tasks land, the `test-analyst` agent reads
this file to (1) check that the implemented tests actually cover every AC the
phase claims (cross-referencing each phase file's "Verifies:" task lines and
"Tests touched/added" tables against the rows below), and (2) assemble the
human test plan -- the residual checks a person must run by hand because they
cannot be (fully) automated. A criterion is "covered" only when a concrete test
asserting it exists at (approximately) the cited path, or its human-verification
procedure has been performed and recorded. None of the 22 ACs may be left
unmapped.

## Conventions

- **Test type**: `unit` = in-crate Rust `#[cfg(test)] mod tests` (no `file_io`,
  AST/equation-text-direct or `TestProject`-builder style); `integration` = Rust
  `tests/*.rs` (here always `src/simlin-engine/tests/simulate_ltm.rs`, which is
  `required-features = ["file_io"]`, compile-and-simulate then assert on
  `Results`); `operational` = run a command and check its exit/output (the
  `cargo test --workspace` budget gate, the pre-commit hook, `scripts/check-docs.py`);
  `human` = a person inspects docs/issues/behavior.
- A single AC may cite multiple rows when coverage is layered (e.g. an
  equation-string unit test in `ltm_augment.rs` plus an end-to-end value check
  in `simulate_ltm.rs`).
- Equation-string assertions compare against `print_eqn`'s canonical form
  (identifiers/element-names lowercased; parsed builtins lowercased; synthesized
  `PREVIOUS` uppercase; single space around binary operators; precedence parens
  reintroduced) -- e.g. `(population[nyc] - PREVIOUS(population[boston])) * 0.01`.
- File paths are the *expected* locations from the phase files; the analyst
  reconciles against where the tests actually landed.
- The `cross_element_ltm` fixture is `test/cross_element_ltm/cross_element.stmx`
  (Region={NYC,Boston}, euler 0..50 dt 1; per-element `population` stock,
  per-element `migration_pressure` aux and `migration_in` flow, A2A `births` and
  `migration_out` flows, scalar `total_population = SUM(population[*])`; **no
  `ltm_results.tsv`** -- structural + simulate-and-assert tests only). Note the
  fixture writes `migration_in[NYC] = MAX(migration_pressure[Boston] * -1, 0)`,
  not the design prose's `MAX(-migration_pressure[Boston], 0)` -- string-match
  assertions must use the `* -1` spelling or compare parsed ASTs.
