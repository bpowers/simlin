# LTM Cross-Element Aggregate Scoring — Phase 4: Arrayed-result reducer support in the reducer link-score machinery

**Goal:** The existing reducer link-score machinery — which today computes per-source-element link scores only for arrayed-source → *scalar*-target reducers (`SUM(pop[*])` → a scalar) — gets extended to handle **partial reduces**: a reducer that collapses only *some* of the source's axes, leaving an arrayed result over the remaining axes (`agg[D1] = SUM(matrix[D1,*])`). This is a prerequisite for Phase 5 (which hoists *every* inlined reducer into an aggregate node — including partial reduces inside an arrayed target).

**Architecture:** `classify_reducer` (`ltm_augment.rs`) recognizes reducers but carries no axis information; `try_cross_dimensional_link_scores` (`db_ltm.rs`, nested in `model_ltm_variables`) early-returns `None` when the target is arrayed, so the partial-reduce path is currently dead — even though `link_score_dimensions` already contains the partial-collapse branch (`to_dims ⊆ from_dims`). We extend the reducer machinery so a `matrix[D1,D2] → agg[D1]` edge produces per-source-element link scores, naming each per `(d1, d2)` pair as a scalar variable `$⁚ltm⁚link_score⁚matrix[d1,d2]→agg[d1]` (`dimensions: vec![]`), consistent with the existing full-reduce per-element naming — which `parse_link_offsets` already handles via its `[`-in-`to` single-passthrough branch, so no parser change is needed.

**Tech Stack:** Rust; `simlin-engine`; `ltm_augment.rs` (`classify_reducer`/`ReducerKind`, `generate_element_to_scalar_equation`, `generate_linear_partial`/`generate_nonlinear_partial`); `db_ltm.rs` (`try_cross_dimensional_link_scores`, `link_score_dimensions`); `ltm_finding.rs` (`parse_link_offsets`); the AST array-shape machinery (`compiler/context.rs::make_dimension_subscripts`, `IndexExpr2`, `BuiltinFn`).

**Scope:** Phase 4 of 6 from `docs/design-plans/2026-05-09-ltm-503-cross-element-agg.md`.

**Codebase verified:** 2026-05-09 (codebase-investigator). **Risk flag (design R3):** the codebase-investigator found that `classify_reducer`/`ReducerKind` carry no axis info and `generate_element_to_scalar_equation` assumes a scalar target — extending both to arrayed results is non-trivial. If during Task 1's TDD the generalization proves much larger than the tasks below describe, STOP and surface to the user (the design explicitly contemplated splitting this phase).

## Context for the implementer (read before starting)

### What "the reducer link-score machinery" is today

When an arrayed variable feeds a *scalar* target through a reducing builtin (`total_pop = SUM(pop[*])`), LTM emits **one scalar link-score variable per source element** — `$⁚ltm⁚link_score⁚pop[nyc]→total_pop`, `$⁚ltm⁚link_score⁚pop[boston]→total_pop`, ... each with `dimensions = vec![]` and an equation that is the ceteris-paribus partial "vary `pop[d]`, hold the others at `PREVIOUS`, see effect on `SUM(pop[*])`". This is done by `try_cross_dimensional_link_scores` (a nested fn inside `model_ltm_variables`) calling `classify_reducer` + `generate_element_to_scalar_equation`. Phase 4 generalizes it to a *partial* reduce: `agg[D1] = SUM(matrix[D1,*])` collapses only the `D2` axis, leaving an arrayed result over `D1` — so for source element `matrix[d1,d2]` the relevant link score targets only `agg[d1]` (not the whole `agg`), and there's one link score per `(d1, d2)` pair.
