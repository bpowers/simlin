# LTM Arrays Hardening — Phase 7 Implementation Plan

**Goal:** Two static-link-polarity fixes in `ltm/polarity.rs`:
- **#502** — per-element graphical functions (`effect[Region] = LOOKUP(curve[Region], dose[Region])`, where `curve` stores one lookup `Table` per region) currently get `Unknown` link polarity because `analyze_link_polarity`'s `Lookup` arm only handles a bare `Expr2::Var` table reference. Fix: extend the `Lookup` arm to handle `Expr2::Subscript` — a `FixedIndex` element (`LOOKUP(curve[NYC], x)`) resolves to an offset and analyzes `vars[curve].tables[offset]`; a Bare-A2A reference (`curve[D]` inside `effect[D] = LOOKUP(curve[D], ..)`) aggregates per-element table monotonicity (agree → that polarity, disagree → `Unknown`, reusing the `Ast::Arrayed` per-element accumulation pattern); a `Wildcard`/`DynamicIndex` subscript → `Unknown`.
- **#492** — `analyze_graphical_function_polarity` uses an absolute epsilon `const EPSILON = 1e-10` on `dy` (the y-delta), which flags lookup tables that are monotone modulo round-trip numeric-import noise as non-monotone. Fix: replace it with a y-range-relative epsilon `epsilon = (1e-6 * (y_max - y_min)).max(1e-12)` (computed from `table.y` — the `Table.y_range` field is module-private to `variable.rs` and not reachable from `ltm/polarity.rs`); the rest of the monotonicity logic unchanged.

**Architecture:** Self-contained in `src/simlin-engine/src/ltm/polarity.rs`. The new `Subscript` arm needs its own tiny inline subscript classifier — `ltm/polarity.rs` is *upstream* of `db_analysis.rs` (the module dependency runs `db_analysis → crate::ltm`, not the reverse) so it cannot call `classify_subscript_shape`/`RefShape` (and `db_ltm_ir.rs` from Phase 1 is in the same `db.rs`-downstream layer) — instead, classify directly on `&[IndexExpr2]` using `Dimension::get_offset` (an element name resolves to an offset; a dimension-name iterator does not). The Bare-A2A case reuses the existing `Ast::Arrayed` fold (`polarity.rs:39-71` — adopt-first-concrete; concrete-disagreement → `Unknown`; an `Unknown`-among-concretes is *ignored*) over the GF source variable's `tables` (which `build_tables` stores one-per-element-in-element-order, with `Table::empty()` placeholders so `tables[offset]` aligns). The epsilon change is a 3-line edit (compute `y_min`/`y_max` via `fold` over `table.y`, then `let epsilon = (1e-6 * (y_max - y_min)).max(1e-12);`).

**Tech Stack:** Rust; the crate's `Expr2`/`IndexExpr2` (`src/ast/expr2.rs`), `BuiltinFn::Lookup` (`src/builtins.rs`), `Variable::Var.tables: Vec<Table>` + `Table { x: Vec<f64>, y: Vec<f64>, .. }` (`src/variable.rs`), `Dimension::get_offset(&CanonicalElementName) -> Option<usize>` (`src/dimensions.rs:128`), `CanonicalElementName` (`src/common.rs:43`).

**Scope:** Phase 7 of 8 (GH #502 + #492). **Independent** of all other phases (Phases 1-6 will have landed but don't touch `ltm/polarity.rs` except Phase 1's `reducer_kind` — and `analyze_link_polarity`'s reducer arms (`Sum`/`Mean`/`Min`/`Max`/`Stddev`/`Rank`) are a *separate* polarity concern Phase 1 explicitly does NOT consolidate, so `polarity.rs` is essentially untouched until this phase).

**Codebase verified:** 2026-05-12 (codebase-investigator report; see "Codebase notes" below).

## Acceptance Criteria Coverage

### ltm-arrays-hardening.AC7: #502 + #492 — per-element graphical-function static polarity; plateau-tolerant GF monotonicity
- **ltm-arrays-hardening.AC7.1 Success:** a model with `effect[Region] = LOOKUP(curve[Region], dose[Region])` where every region's `curve` table is monotone increasing — the link `dose[Region] -> effect[Region]` (and any loop through it) gets `Positive` static polarity (composed with the argument's polarity), not `Unknown`.
- **ltm-arrays-hardening.AC7.2 Success:** the same model where the regions' tables disagree on monotonicity (some increasing, some decreasing) — the link polarity is `Unknown` (the per-element aggregation correctly reports disagreement).
- **ltm-arrays-hardening.AC7.3 Success:** `LOOKUP(curve[NYC], x)` (a `FixedIndex` per-element-GF reference) gets the static polarity of NYC's specific table (`tables[offset_of(nyc)]`).
- **ltm-arrays-hardening.AC7.4 Success:** a lookup table whose y-values are monotone non-decreasing modulo round-trip numeric noise (e.g. one segment dips by ~1e-7 against a unit-scale y-range) is classified `Positive`, not `Unknown` — the y-range-relative epsilon tolerates it.
- **ltm-arrays-hardening.AC7.5 Failure / Edge:** a genuinely non-monotone table (a real direction change, not noise) still returns `Unknown`; a constant table still returns `Unknown`.

## Note on AC7.5 / the `Ast::Arrayed` fold semantics

The existing `Ast::Arrayed` per-element polarity fold (and the `fold_per_element_table_polarity` helper this phase adds, which mirrors it) is "adopt-first-concrete; concrete-disagreement → `Unknown`; an `Unknown`-among-concretes is ignored (kept as the running concrete polarity)". So a per-element graphical function where *one* element's table is genuinely non-monotone (its `analyze_graphical_function_polarity` is `Unknown`) but the others all agree on `Positive` yields `Positive` for the link, not `Unknown`. AC7.2 ("the regions' tables *disagree on monotonicity* — some increasing, some decreasing") is the per-element-GF disagreement case (`Positive` vs `Negative` ⇒ `Unknown`); AC7.5 ("a genuinely non-monotone table still returns `Unknown`") is about `analyze_graphical_function_polarity` *itself* (Task 1's domain) and about a *scalar* table — not "any non-monotone element among monotone ones flips the link to `Unknown`" (that would be a different fold, changing the established `Ast::Arrayed` semantics — out of scope). The phase file's tests follow this faithful reading.

**Design-owner confirmation (before writing the AC7.5 / per-element-GF-link assertion):** if you are about to rely on this reframed reading — i.e. AC7.5 = the scalar-table case (Task 1's domain) and "a non-monotone element among monotone ones still yields a concrete link polarity" per the established `Ast::Arrayed` fold — confirm the intent with the design owner via AskUserQuestion. The faithful reading is the right one (reusing the existing fold is exactly what the design's "Components" section says to do), but if the design owner intended the stricter "any non-monotone element ⇒ `Unknown` link", that's a separate, broader change (it would alter the established `Ast::Arrayed` polarity semantics for *all* arrayed variables, not just per-element GFs) and must be flagged, not silently adopted. (Batch this with Phase 5's `(m-1)!/2`-vs-AC5.2 open question and Phase 6's AC6.2-wording note if convenient.)
