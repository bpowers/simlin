# LTM Arrays Hardening — Phase 6 Implementation Plan

**Goal:** STDDEV's per-element ceteris-paribus link-score partial currently falls back to a degenerate delta-ratio (`generate_nonlinear_partial`'s STDDEV/RANK catch-all returns `target_q.to_string()`, which makes the surrounding link-score formula `|Δtarget/Δtarget| · sign(...)` ≈ degenerate-1/0). Replace it with the true analytic per-element partial: for source element `d`, build `sqrt((sum_i (s_i' - m)^2) / N)` with `s_i' = s[d]` if `i == d` else `PREVIOUS(s[i])`, and `m = (sum_i s_i') / N` (matching the engine's STDDEV, which is population variance `/N`). RANK keeps the delta-ratio with a documented in-code justification (RANK is an order statistic — non-differentiable, array-argument-only, and unreachable via real models since RANK returns an array and can't be a scalar/A2A reducer RHS), pinned by a test so it is an explicit choice, not a silent fallback. SUM / 1-arg MEAN / 1-arg MIN / 1-arg MAX partials are unchanged.

**Architecture:** The change is confined to `ltm_augment.rs`: the `_ =>` arm of `generate_nonlinear_partial` (the STDDEV/RANK catch-all) becomes a `match reducer_name { "STDDEV" => <build the unrolled variance formula>, "RANK" => <delta-ratio with a justification comment>, _ => unreachable!() }`. The unrolled STDDEV formula is built by string concatenation (`format!`), exactly like the existing MIN/MAX unroll over `all_elements` — `PREVIOUS(s[i])` is emitted as the literal string `PREVIOUS({source_q}[{elem}])`, which the engine's `builtins_visitor` transparently rewrites into a synthesized scalar helper aux, so there is **no** "PREVIOUS requires a variable reference" wrinkle (proven by the existing `test_cross_dim_min_expansion` / `test_cross_dim_max_expansion`). `build_element_reducer_link_score`'s outer `Δtarget = 0` guard means the link score is 0 whenever STDDEV genuinely doesn't change — so the fix is distinguished from the status quo not by "STDDEV invariant ⇒ scores ≈ 0" (true for both) but by "multiple elements changing by different amounts ⇒ the analytic partial isolates `d`'s contribution while the delta-ratio conflated them" (the AC6.3 hand-calc).

**Tech Stack:** Rust; string-built equation text (re-parsed by the engine); `BuiltinFn::Sqrt` ("sqrt", parses fine) and `BinaryOp::Exp` (`^`, binds tighter than `+/-/*//`); the consolidated `ReducerKind` from Phase 1 (re-exported into `ltm_augment` from `ltm_agg`).

**Scope:** Phase 6 of 8 (GH #483). **Depends only on Phase 1's reducer-table consolidation** (cosmetic — `ReducerKind` moves to `ltm_agg.rs` and is re-exported; `classify_reducer` still returns `Option<(ReducerKind, &'static str, bool)>`). **Not** a behavioral dependency on Phase 4 — the agg-routing infra that lands the STDDEV partial on the `source[d] → agg` (or variable-backed `source[d] → target`) `is_bare = true` path is already in `main` (the "Phase 5 of the cross-element-aggregate-scoring design" work, `ltm_agg.rs:8`); Phase 4 (sliced-reducer hoisting) doesn't change STDDEV.

**Codebase verified:** 2026-05-12 (codebase-investigator report; see "Codebase notes" below).

## Acceptance Criteria Coverage

### ltm-arrays-hardening.AC6: #483 — analytic STDDEV (and RANK) ceteris-paribus partials
- **ltm-arrays-hardening.AC6.1 Success:** for a model where `total = STDDEV(s[*])` feeds back into `s`, the per-element link score `$⁚ltm⁚link_score⁚s[d]→total` is the analytic ceteris-paribus partial (the unrolled variance formula holding `s[d]` live, the other elements at `PREVIOUS`), not `(total - PREVIOUS(total)) / (s[d] - PREVIOUS(s[d]))`.
- **ltm-arrays-hardening.AC6.2 Success:** under uniform scaling of all elements each step (all `s[i]` multiplied by the same factor), STDDEV does not change, so the per-element link scores `s[d] -> total` are ~0 at every step >= 1 — the pre-fix delta-ratio reported spurious nonzero values. *(See note: "uniform scaling by a factor" is mathematically a STDDEV-changing operation; the AC's intent is "a regime where STDDEV is constant" — the fixture must use a constant/additive uniform per-element flow, not a proportional one.)*
- **ltm-arrays-hardening.AC6.3 Success:** the STDDEV link score matches a hand calculation at >= 1 timestep within 1e-6.
- **ltm-arrays-hardening.AC6.4 Edge:** RANK — either the analytic per-element partial is implemented (a test pins it), or RANK keeps the delta-ratio with a documented in-code justification (RANK is an order statistic: non-differentiable, array-argument-only) and a test pins that explicit choice (so it is not a silent fallback).
- **ltm-arrays-hardening.AC6.5 Success (no regression):** SUM / MEAN / 1-arg MIN / 1-arg MAX per-element partials are unchanged (the existing `test_generate_*_equation` / `test_generate_reduced_*_equation` tests pass).

## Note on AC6.2's wording

The design's AC6.2 says "under uniform scaling of all elements each step (all `s[i]` multiplied by the same factor), STDDEV does not change" — that is mathematically incorrect (`STDDEV(c·x) = |c|·STDDEV(x) ≠ STDDEV(x)` for `c ≠ 1`); only an *additive* uniform shift (`STDDEV(x + k) = STDDEV(x)`) leaves STDDEV invariant. The phase file interprets AC6.2's *intent* as "a regime where STDDEV is constant ⇒ per-element link scores are ~0" and builds the fixture with a constant/additive uniform per-element flow accordingly. Additionally: in any STDDEV-invariant regime the `Δtarget = 0` guard in `build_element_reducer_link_score` zeros the link score *regardless of which partial is used*, so AC6.2 alone doesn't distinguish the analytic fix from the pre-fix delta-ratio — the load-bearing verification is AC6.3's hand-calc (which also asserts the analytic value differs from the pre-fix value).

**Design-owner confirmation (before writing the AC6.2 fixture/assertion):** if you are about to rely on this reframed reading (constant/additive flow rather than the literally-stated "multiplied by the same factor"), confirm the intent with the design owner via AskUserQuestion. The faithful reading — "a regime where STDDEV is constant" — is almost certainly correct, but the literal AC text is mathematically wrong, so flag it rather than silently proceeding.
