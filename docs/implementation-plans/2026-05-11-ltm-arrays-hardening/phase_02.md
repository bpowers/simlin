# LTM Arrays Hardening — Phase 2 Implementation Plan

**Goal:** Make A2A (apply-to-all) feedback loops carry element-level `Loop::stocks` and resolve a real cycle partition *per slot*, so two disconnected A2A feedback subsystems over different dimensions stop cross-normalizing in the post-simulation relative-loop-score computation.

**Architecture:** `Loop::stocks` becomes element-level (`{var}[{elem}]` over the loop's dimension element space) for arrayed loops, exactly as the cross-element/mixed loop branches already do — the A2A branches just adopt the same strip-and-rebuild pattern. `LtmVariablesResult.loop_partitions` changes from `HashMap<String, Option<usize>>` to `HashMap<String, Vec<Option<usize>>>` (one entry per A2A-loop slot; length 1 for scalar/cross-element loops). `partition_for_loop` returns one partition per slot. The post-simulation normalizers (`compute_rel_loop_scores` / `compute_rel_loop_scores_per_element`) — which already have most of the per-element/per-slot machinery from PR #463 — switch their grouping key from per-loop to `(partition_index, slot)` and read the partition per slot from the new vector. The FFI ripple is small: `loop_partitions` is purely internal to `libsimlin::SimState` (never crosses the C header / `@simlin/engine` / `pysimlin` boundary — those carry loop `variables` from links, never stocks/partitions); the touch list is `libsimlin/src/{lib,simulation,analysis}.rs` (a field type, a type alias, the slot-k partition lookup in `ensure_denom_for_element`) plus one `layout/mod.rs` call site.

**Tech Stack:** Rust; salsa (the `LtmVariablesResult` is a `#[salsa::tracked(returns(ref))]` result — `salsa::Update` must be derivable for the new shape, which it is for `Vec<Option<usize>>`); `libsimlin` C-FFI crate (cbindgen-generated `simlin.h` — adding an *optional* per-slot accessor is the only header change, and only if AC2.5's additive accessor is implemented).

**Scope:** Phase 2 of 8 (GH #487). No dependency on Phase 1 (independent). **Possible split:** the design (risk R2) allows splitting into "engine-side" + "FFI/layout surface" if the test churn balloons; the codebase investigation found the FFI footprint small enough to keep it as one phase — **default to one phase; only split if the test churn proves unmanageable, in which case file a follow-up implementation plan for the FFI half.**

**Codebase verified:** 2026-05-12 (codebase-investigator report; see "Codebase notes" below).

## Acceptance Criteria Coverage

### ltm-arrays-hardening.AC2: #487 — partition-correct A2A loop normalization
- **ltm-arrays-hardening.AC2.1 Success:** a model with two disconnected A2A feedback subsystems over *different* dimensions (e.g. `pop[Region] -> births[Region] -> pop[Region]` and `widgets[Product] -> production[Product] -> widgets[Product]`) — each loop's relative loop score normalizes against the sum of absolute loop scores within its own cycle partition, not a shared bucket; the two loops do not cross-normalize. Verified by a hand calculation matching each per-loop relative score and differing from the pre-fix pooled value.
- **ltm-arrays-hardening.AC2.2 Success:** an A2A loop over an N-element dimension has N entries in `loop_partitions` (one per slot); for a pure-A2A loop over an element-wise-uncoupled dimension those are N distinct partition indices; for an element-wise-coupled dimension they coincide; scalar and cross-element loops have exactly one entry.
- **ltm-arrays-hardening.AC2.3 Success:** `Loop::stocks` for an A2A loop is the element-level `{var}[{elem}]` list (not the variable-level name); the `Loop` struct's granularity docstring states this; `enrich_with_module_stocks`, JSON SDAI relationships, and layout feedback metadata still produce correct output for A2A loops.
- **ltm-arrays-hardening.AC2.4 Success (no regression):** a purely scalar model's loops and relative scores are unchanged; a model with a single A2A feedback subsystem has the same relative loop score it had before (the pre-fix single-`None`-loop value coincided with the correct partition-local value).
- **ltm-arrays-hardening.AC2.5 Success:** the FFI surface (`libsimlin`'s `loop_partitions` snapshot, the generated C header, `@simlin/engine`, `pysimlin`) exposes the per-slot partitions with a slot-0 convenience for callers that want a single value; a round trip through the FFI preserves them.

- `partition_for_loop(&self, loop_item: &Loop) -> Vec<Option<usize>>`:
  - If `loop_item.dimensions.is_empty()` (scalar or cross-element/mixed): a singleton — the partition of the loop, looked up via `find_map` over `loop_item.stocks` in `self.stock_partition` (the element-level stock names like `"pop[nyc]"` are the keys in the element-level `CyclePartitions`; for a scalar loop they're plain names).
