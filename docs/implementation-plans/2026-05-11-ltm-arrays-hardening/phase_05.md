# LTM Arrays Hardening — Phase 5 Implementation Plan

**Goal:** A reducer in a feedback loop over a >8-element dimension currently yields **zero** cross-element-through-aggregate loops (the hard `petals.len() > MAX_AGG_PETALS (=8) -> continue` drop), and the recovery enumerates only the `2^k`-bitmask of petal *subsets* (one ordering per subset). Fix: replace the hard drop with a deterministic petal-priority ordering plus an overall loop-count *budget*; when the budget is hit, recovery stops, the result carries a `TruncatedByBudget`-style flag, and a `CompilationDiagnostic` `Warning` is emitted (mirroring the auto-flip-to-discovery gate). And: replace the per-subset enumeration with distinct-*cyclic-ordering* enumeration — for each chosen petal subset of size m ≥ 2, all cyclic orderings (first petal pinned to kill rotations, mirror reversals skipped — `1` ordering for m = 2, `(m-1)!/2` for m ≥ 3, via Heap's algorithm or `itertools::permutations`), under the same budget.

**Architecture:** `recover_cross_agg_loops` (private, `db_ltm.rs`) currently: extracts "petals" (agg-touching elementary circuits `agg -> ... -> agg`, rotated to start at the agg, with the non-agg nodes as "internal nodes"), groups them by agg, then for each agg with `2 <= petals.len() <= 8` enumerates the `2^k` subsets, keeps the pairwise-disjoint ones (≥2 petals, no internal-node overlap), and stitches each subset into one `Loop` in the petals' discovery order. Phase 5: drop the `<= 8` cap; sort an agg's petals by a deterministic priority (e.g. fewest internal nodes first, then a stable tiebreaker — smaller petals combine into more loops, and a deterministic order makes the truncation reproducible); walk subsets (or rather: for each disjoint subset, enumerate its distinct cyclic orderings); count emitted loops against a budget constant (`#[cfg(test)]`-overridable, replacing/supplementing `MAX_AGG_PETALS`); when the budget is exhausted, stop, set a truncation flag, and `.accumulate(db)` a `Warning`. The truncation flag rides on `LtmVariablesResult` (a new `bool` field — `salsa::Update` is auto-derivable for `bool`) since `recover_cross_agg_loops` is three private functions below `model_ltm_variables` and the cleanest place to surface it is the result struct (the `Warning` is emitted directly from inside the call chain, which is reachable only from the tracked `model_ltm_variables`, so `.accumulate(db)` works). `recover_agg_hop_polarities` is unchanged.

**Tech Stack:** Rust; salsa (`CompilationDiagnostic` accumulator; `LtmVariablesResult` gains a `bool` field); a permutation enumerator — either hand-rolled Heap's algorithm (the crate convention leans toward hand-rolling small algorithms — cf. `lex_smallest_rotation_start` in `ltm/mod.rs`) or add `itertools` as a dependency (heavier; reviewable). **No `MAX_LTM_TOTAL_CIRCUITS` gate exists** — the only structural gate is `MAX_LTM_SCC_NODES = 50` — so the new budget constant must be `#[cfg(test)]`-overridable and the new tests must use a tiny override + a small fixture (per `docs/dev/rust.md#test-time-budgets` — the PR #461 cautionary tale).

**Scope:** Phase 5 of 8 (GH #515). **Interacts with Phase 4** (a sliced agg can also sit in a feedback loop — `recover_cross_agg_loops`'s petal extraction must handle a *subscripted* agg node consistently; ordered after Phase 4 for the combined test surface) but has no mechanical dependency on it.

**Codebase verified:** 2026-05-12 (codebase-investigator report; see "Codebase notes" below).

## Acceptance Criteria Coverage

### ltm-arrays-hardening.AC5: #515 — budgeted cross-agg loop recovery; cyclic-ordering enumeration
- **ltm-arrays-hardening.AC5.1 Success:** a reducer in a feedback loop over a >8-element dimension (so >8 disjoint petals through one agg) recovers a non-empty, budgeted set of cross-element-through-agg loops (not zero, as today); the result carries a `TruncatedByBudget` flag (and a `CompilationDiagnostic` `Warning` is also emitted) so the model author knows the loop list is incomplete.
- **ltm-arrays-hardening.AC5.2 Success:** for a model with k = 3 disjoint petals through one agg, all distinct cyclic orderings are recovered as distinct directed cycles (`A->p1->A->p2->A->p3->A` and `A->p1->A->p3->A->p2->A` are both present) within the loop budget — not just the single per-subset ordering today's `2^k`-bitmask enumeration produces.
- **ltm-arrays-hardening.AC5.3 Success (no regression):** a model with <= 8 petals and at most 2 petals per recovered loop produces the same cross-agg loops as before (the per-subset enumeration for m = 2 has exactly one cyclic ordering, which equals today's output).
- **ltm-arrays-hardening.AC5.4 Edge:** the recovered cyclic orderings of one petal subset share a loop score (the product over the same edge set); a test confirms their `loop_score` series are equal even though they are distinct loops by the directed-edge-sequence identity rule.
