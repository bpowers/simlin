# LTM Arrays Hardening — Phase 3 Implementation Plan

**Goal:** Two arrayed-reference shapes the pre-#520 code could not represent become correct:
- **#511** — an A2A equation that references an arrayed dependency by its *iterated dimension* (`growth[D1,D2] = row_sum[D1] * c`, `row_sum` over `D1`, `growth` over `D1×D2`) currently misclassifies the subscript as `DynamicIndex`, the link-score partial PREVIOUS-wraps a `Subscript`, and codegen rejects it with "PREVIOUS requires a variable reference after helper rewriting". Fix: classify an iterated-dimension subscript as `RefShape::Bare` (a same-element-on-shared-dims reference) in the #520 IR, and normalize iterated-dimension subscripts on source references to bare form in the partial-equation `Expr0` before PREVIOUS-wrapping (the model equation is untouched, so simulation still evaluates `row_sum[d1]`).
- **#510** — a disjoint-dimension arrayed→arrayed edge with *per-element target equations* (`target[D1,D2]` whose `<element subscript="...">` equations reference `source[D3]`, D3 disjoint from D1/D2) currently degenerates to a silent scalarized stand-in. Fix: emit one link-score variable per distinct referenced source element (`$⁚ltm⁚link_score⁚source[m]→target`, ...), each an `Equation::Arrayed` over `target`'s dims with the partial holding `source[m]` live in the slots that reference it and the trivial-zero guard form elsewhere; `link_score_dimensions` returns `target`'s dims; and a clear compile-time `Warning` diagnostic for a genuinely-unscoreable edge (e.g. a `DynamicIndex` source into such a target) instead of silent scalarization.

**Architecture:** #511 is a small classifier change in two places — the #520 IR's `model_ltm_reference_sites` walker (recognize "this subscript's indices are exactly the target equation's iterated dimension names, in matching positions" ⇒ `RefShape::Bare`) and the partial-equation builder (`build_partial_equation_shaped` / `wrap_non_matching_in_previous` / `classify_expr0_subscript_shape` get the target's iterated-dim names threaded in and normalize `row_sum[D1]` → `row_sum` in the partial `Expr0`). The element-graph side then needs nothing new — `emit_edges_for_reference`'s `RefShape::Bare`-arrayed arm already does the `expand_same_element` same-element-on-shared-dims projection. #510 is a new branch in `try_cross_dimensional_link_scores` (or a sibling fn called from `emit_link_scores_for_edge`) that, for a per-element-equation arrayed target referencing source elements of a disjoint dimension, emits one `$⁚ltm⁚link_score⁚source[m]→target` per distinct referenced source element via `build_arrayed_link_score_equation` (which already handles `FixedIndex` source into an `Ast::Arrayed` target per slot, with the trivial-zero guard) — plus `link_score_dimensions` returning `target`'s dims for that edge (it returns `vec![]` today, which is what triggers `retarget_ltm_equation_dims` → `scalarize_ltm_equation`), plus a `CompilationDiagnostic` `Warning` (the established `.accumulate(db)` pattern) for the unscoreable case.

**Tech Stack:** Rust; the crate's `Expr0` AST (`src/ast/expr0.rs`) for the partial-equation builder, `Expr2` (`src/ast/expr2.rs`) for the IR walker; `datamodel::Equation::{Scalar,ApplyToAll,Arrayed}`; salsa (the IR `model_ltm_reference_sites` from Phase 1; `CompilationDiagnostic` accumulator).

**Scope:** Phase 3 of 8 (GH #511 + #510). **Depends on Phase 1** (the #520 IR — the iterated-dimension classification lives there). Independent of Phase 2.

**Codebase verified:** 2026-05-12 (codebase-investigator report; see "Codebase notes" below).

## Acceptance Criteria Coverage

### ltm-arrays-hardening.AC3: #511 + #510 — iterated-dimension subscripts; disjoint-dim arrayed→arrayed link scores
- **ltm-arrays-hardening.AC3.1 Success:** `growth[D1,D2] = row_sum[D1] * c` (with `row_sum` over `D1`, `growth` over `D1 x D2`) + LTM enabled compiles; the element graph has the same-element-on-shared-dims projection `row_sum[d1] -> growth[d1,*]` (not the full cross-product); `$⁚ltm⁚link_score⁚row_sum→growth` is emitted as the Bare partial (`row_sum` held live, no spurious `SUM(...)` source-ref, no `PREVIOUS`-wrapped `Subscript`) and simulates without the `"PREVIOUS requires a variable reference after helper rewriting"` error.
- **ltm-arrays-hardening.AC3.2 Success:** a feedback loop through such an edge (`... -> row_sum[D1] -> growth[D1,D2] -> ... -> row_sum[D1]`) is enumerated and its loop-score equation references `$⁚ltm⁚link_score⁚row_sum→growth`; the loop score matches a hand calculation at >= 1 timestep within 1e-6.
- **ltm-arrays-hardening.AC3.3 Success:** a disjoint-dim arrayed→arrayed model with per-element target equations (`target[D1,D2]` whose `<element subscript="...">` equations reference `source[D3]`, D3 disjoint from D1/D2) emits one link-score variable per distinct referenced source element (`$⁚ltm⁚link_score⁚source[m]→target`, ...), each an `Equation::Arrayed` over `D1 x D2` whose partial holds `source[m]` live in the slots that reference it and is the trivial-zero guard form elsewhere — not a single scalar stand-in; `link_score_dimensions` returns `target`'s dims for this edge.
- **ltm-arrays-hardening.AC3.4 Failure:** an edge that is genuinely unscoreable this way (e.g. a `DynamicIndex` source into a disjoint-dim arrayed target) produces a clear compile-time `Warning` diagnostic naming the unscoreable edge, not a silent scalarized stand-in.
- **ltm-arrays-hardening.AC3.5 Edge:** a mapped-dimension iterated subscript (`x` over `Region`, target over `State`, a `State -> Region` mapping, `x[State]` referenced) is handled the same way a bare `x` reference into that target is handled today — no new dimension-mapping behavior, no regression.

**Implementation notes:**
- "Position matches" — be precise about what counts. The clean rule: the subscript `source[d_0, d_1, ...]` is `Bare` iff `source`'s declared dimension order is `[D_0, D_1, ...]` (possibly a prefix/subset of the target's dims), and for each `i`, `d_i` is either the same dimension name as `D_i` or a dimension that maps to `D_i`, *and* `d_i` is one of the target equation's iterated dimensions. (This is exactly "the reference iterates over the target's dimension space and reads the same element of the source" — the thing `expand_same_element` then projects.) A subscript like `row_sum[D2]` inside `growth[D1,D2]` where `row_sum` is over `D1` would NOT be `Bare` (position mismatch) — keep that `DynamicIndex` for now (it's a genuine cross-element reference; out of scope here).
