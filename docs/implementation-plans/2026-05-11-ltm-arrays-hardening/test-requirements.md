# LTM Arrays Hardening — Test Requirements

Maps each acceptance criterion from `docs/design-plans/2026-05-11-ltm-arrays-hardening.md` to an automated test (type + expected test file path) or to documented human verification, rationalized against the implementation decisions made in `phase_01.md` … `phase_08.md`.

**Scope of "automated":** the entire cluster is Rust engine work plus FFI-consumer updates; almost everything is verifiable by automated unit/integration tests. The few human-verification items are: GitHub issue closure and epic checklist edits (#488), and doc-content review (the prose in `docs/design/ltm--loops-that-matter.md` / `docs/reference/ltm--loops-that-matter.md` is reviewed, not test-asserted — but the doc-link integrity is automated via `scripts/check-docs.py`).

**Test-file conventions** (from the codebase-investigator passes): engine unit tests live in inline `#[cfg(test)] mod ..._tests` blocks or `#[cfg(test)] #[path = "..._tests.rs"] mod ...;`-included files under `src/simlin-engine/src/`; integration tests in `src/simlin-engine/tests/` (the LTM ones — `simulate_ltm.rs`, `wrld3_ltm_panic.rs` — are `required-features = ["file_io"]`, run with `cargo test -p simlin-engine --features file_io --test simulate_ltm`); `libsimlin` tests in `src/libsimlin/tests/`. New IR tests go in `src/simlin-engine/src/db_ltm_ir_tests.rs` (wired via `#[cfg(test)] #[path] mod db_ltm_ir_tests;` in `db_ltm_ir.rs`). There is no snapshot framework; the one checked-in golden trace is `test/logistic_growth_ltm/ltm_results.tsv` (compared at 5% rel tol by `tests/simulate_ltm.rs::ensure_ltm_results`); "byte-unchanged golden data" ⇒ all LTM tests pass with no test-file edits.
