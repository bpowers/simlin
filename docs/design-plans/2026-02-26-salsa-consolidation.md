# Salsa Consolidation Design

## Summary

Simlin's simulation engine maintains two parallel compilation paths: a monolithic
bytecode compiler that rebuilds the whole model each time, and a salsa-based
incremental compiler that tracks fine-grained data dependencies so only affected
parts recompile. The incremental path is fully operational for ordinary models,
but three areas still fall back to the monolithic path: LTM analysis, patch
error validation, and internal callers (auto-layout, CLI). This design
eliminates those fallbacks, making the incremental compiler the sole production
path, then deletes the monolithic path entirely.

The approach proceeds in six phases. First, `PREVIOUS` and `INIT` -- time-access
functions heavily used by LTM -- are converted from heavyweight sub-model
instantiations into single VM opcodes, reducing the per-call overhead from 5
variable slots and a module dispatch to a single opcode. With that
simplification, LTM's existing tracked analysis functions are connected to the
incremental compilation pipeline through a parallel compilation path that
appends LTM-specific bytecode fragments without disturbing the main causal
graph. Simultaneously, the engine's error reporting is consolidated onto the
salsa accumulator mechanism, removing the parallel struct-field error path.
Once all callers are migrated, the old monolithic compiler is deleted. See
[the original incremental compilation design](../design/2026-02-21-incremental-compilation.md)
for foundational context.

## Definition of Done

1. **LTM fully incrementalized**: `simlin_sim_new` with `enable_ltm=true` uses the salsa-incremental compilation path. LTM synthetic variables (link scores, loop scores, relative scores, module composites) are generated by existing tracked functions and compiled through the same `assemble_simulation` pipeline as regular variables. The monolithic `with_ltm()` + `compile_simulation()` fallback is eliminated.

2. **Patch error checking uses incremental path only**: `apply_patch` no longer runs the monolithic `compile_simulation` for error detection. All compilation errors (including simulation-level errors like `BadTable`, `EmptyEquation`, `MismatchedDimensions`) are surfaced through the salsa-incremental path with fine-grained error codes. The dual error path (struct fields + accumulators) is consolidated to accumulators only.

3. **Old bytecode compilation path deleted**: `compile_project` (the free function), `Module::compile()`, `Simulation::compile()`, and `build_metadata` are removed. All bytecode compilation goes through `compile_project_incremental` / `assemble_simulation`. `SimlinDb` is threaded to all callers that need simulation results (layout, analysis, LTM methods).

4. **Interpreter preserved**: `Simulation::new()` + `Simulation::run_to_end()` (AST-walking interpreter) remains as a reference implementation. `Module::new` remains (it builds the `Expr` ASTs the interpreter needs).

5. **All existing tests pass**: simulation tests, LTM tests, and integration tests produce identical numerical results.

## Glossary

- **salsa**: An incremental computation framework for Rust. Functions annotated as "tracked" are memoized; when an input changes, only transitively affected computations re-run. Basis for `SimlinDb`. See [original design glossary](../design/2026-02-21-incremental-compilation.md#glossary) for foundational salsa terms.
- **LTM (Loops That Matter)**: An analysis technique that identifies causal feedback loops in a model and computes per-link and per-loop influence scores so the diagram can highlight dominant loops dynamically during simulation.
- **LTM synthetic variables**: Auxiliary variables generated automatically by LTM analysis (one per link or loop). Not present in the user's model source; derived from the causal graph and compiled through a dedicated parallel path.
- **link score / loop score / relative score**: Numerical measures produced by LTM. A link score quantifies the contribution of a single causal edge; loop and relative scores aggregate across whole feedback loops.
- **`LoadPrev` / `LoadInitial`**: New VM opcodes introduced by this design. `LoadPrev` reads a variable's previous-timestep value (from `curr[off]`). `LoadInitial` reads the value captured at t=0.
- **stdlib dynamic module**: A built-in function implemented as a synthetic sub-model with internal variables. SMOOTH, DELAY, and TREND remain as stdlib modules; PREVIOUS and INIT are promoted to builtins.
- **`compile_ltm_var_fragment`**: New tracked function that compiles LTM synthetic variable equations to symbolic bytecode fragments, keyed on `(LtmLinkId, SourceModel, SourceProject)` for per-link caching.
- **`CompilationDiagnostic`**: The salsa accumulator type for collecting compilation errors and warnings. Extended by this design to be the sole error source, replacing struct-field errors.
- **discovery mode**: An LTM mode that compiles scores for every causal link simultaneously, used when the UI shows all loop influences rather than a selected subset.
- **`LtmLinkId`**: A salsa interned type identifying a specific causal link (from-variable, to-variable pair) for per-link LTM caching.

## Architecture

### PREVIOUS and INIT as Builtin Opcodes

`PREVIOUS(x)` and `INIT(x)` are currently stdlib dynamic modules -- each call
instantiates a sub-model with 5 variables (input, initial_value, 2 flows, 1
stock). LTM link score equations use 6+ PREVIOUS calls per variable, creating
~30 data buffer slots and 6 module dispatches per link score at runtime.

Both become VM intrinsic opcodes:

- **`LoadPrev { off: VariableOffset }`** reads `curr[module_off + off]` -- the
  value at the start of this timestep, which equals the previous timestep's end
  value. The VM's step cycle copies `next` into `curr` at the start of each
  timestep, so `curr` always holds prior-step results during dt-phase
  evaluation. This matches the PREVIOUS module's stock-drain-refill semantics
  exactly.

- **`LoadInitial { off: VariableOffset }`** reads from the frozen initial-value
  buffer captured at t=0. The VM already stores initial values; this opcode
  accesses them directly.

Both opcodes get symbolic counterparts (`SymLoadPrev`, `SymLoadInitial`) using
`SymVarRef` instead of raw offsets, fitting into the existing symbolic bytecode
layer in `src/simlin-engine/src/compiler/symbolic.rs`.

The parser maps `PREVIOUS(ident)` and `INIT(ident)` to new `BuiltinFn` variants
instead of triggering `BuiltinVisitor`'s module instantiation path in
`src/simlin-engine/src/builtins_visitor.rs`. `previous` and `init` are removed
from `stdlib::MODEL_NAMES` in `src/simlin-engine/src/stdlib.gen.rs`.

### LTM Parallel Compilation Path

With PREVIOUS as a builtin, LTM synthetic variables are pure scalar arithmetic
expressions -- no module instantiation, no arrays, no graphical functions. They
are compiled through a dedicated parallel path rather than being injected as
`SourceVariable` inputs.

**Why parallel, not unified:** Salsa inputs (`SourceVariable`) cannot be created
inside tracked functions -- they require `&mut SimlinDb`. LTM synthetic
variables are derived from tracked functions (`model_ltm_synthetic_variables`
etc.). Injecting them as `SourceVariable` inputs would create circularity:
`model_causal_edges` reads `model.variables(db)`, so adding LTM vars to the
variable list would pollute the causal graph with spurious edges from
LTM-generated PREVIOUS references.

The parallel path consists of:

- **`compile_ltm_var_fragment`** -- a new tracked function keyed on
  `(LtmLinkId, SourceModel, SourceProject)`. Calls `link_score_equation_text`
  (already tracked and cached per-link), parses, lowers, and emits a
  `CompiledVarFragment` with symbolic bytecode. Skips all module/stock/array
  machinery since LTM vars are always scalar auxes. Loop score and relative
  loop score variables get analogous tracked compilation functions.

- **`compute_layout` extension** -- a third section after implicit variables,
  iterating sorted LTM synthetic variable names from
  `model_ltm_synthetic_variables` (or `model_ltm_all_link_synthetic_variables`
  for discovery mode). Each LTM var gets 1 slot (scalars). Layout only
  recomputes when the set of LTM vars changes.

- **`assemble_module` third pass** -- after compiling explicit and implicit
  variables, iterates LTM synthetic variables and calls
  `compile_ltm_var_fragment` for each. LTM fragments are appended to
  `runlist_flows`. Since `LoadPrev` reads `curr[off]`, LTM vars have no
  dt-phase ordering constraints with regular variables.

**Always-on behavior with opt-out:** An `ltm_enabled: bool` flag on
`SourceProject` (salsa input) controls whether `compute_layout` and
`assemble_module` include the LTM pass. When disabled (e.g., arrayed models
where LTM is not yet supported), zero overhead -- no extra slots, no extra
fragments. LTM tracked functions (causal graph, loop circuits, link score
equations) remain callable regardless of the flag for analysis/introspection.

### Error Accumulator Consolidation

The `CompilationDiagnostic` salsa accumulator in
`src/simlin-engine/src/db.rs` becomes the sole error source, replacing
dual error paths (struct fields on `engine::Project` / `ModelStage1` /
`Variable` + accumulators).

Extended coverage:

- **Per-variable compilation errors** with specific codes (`BadTable`,
  `EmptyEquation`, `MismatchedDimensions`, `TodoArrayBuiltin`, etc.) --
  `compile_var_fragment` already generates these but currently returns `None`
  for failures. Errors are accumulated with their specific codes preserved.

- **Assembly-level errors** from `assemble_module` and `assemble_simulation` --
  model not found, circular dependencies, missing fragments.

- **Unit inference** moves into the salsa pipeline as a tracked function (e.g.,
  `check_model_units`) that accumulates `DiagnosticError::Unit` warnings. This
  replaces the `run_default_model_checks` callback in `Project::base_from`.

- **VM validation** as a separate post-compilation step: `Vm::new(compiled)` is
  called after `compile_project_incremental` to validate bytecode structure.
  Failures are reported alongside accumulated diagnostics.

`DiagnosticError` gains an `Assembly(String)` variant for simulation-level
errors and a severity level (error vs warning) to distinguish blocking errors
from unit warnings.

### Replacing `engine::Project` in `apply_patch`

`SimlinProject` in `src/libsimlin/src/lib.rs` changes its `project:
Mutex<engine::Project>` field to `datamodel: Mutex<datamodel::Project>`. The
salsa database IS the compiled state; the `engine::Project` type is no longer
stored across FFI calls.

New `apply_patch` flow in `src/libsimlin/src/patch.rs`:

1. Snapshot `datamodel::Project` under lock (same as today).
2. Apply engine patch to staged datamodel (same as today).
3. `sync_from_datamodel_incremental` syncs salsa inputs (same as today).
4. `compile_project_incremental(&db, project, "main")` triggers the full salsa
   pipeline. Errors accumulate via `CompilationDiagnostic`.
5. If compilation succeeds, `Vm::new(compiled)` validates bytecode.
6. `collect_all_diagnostics(&db, &sync)` reads accumulated errors. Accept/reject
   logic uses diagnostics instead of walking struct fields.
7. Rollback or commit with same lock protocol. Committed state is
   `datamodel::Project` + `SimlinDb`.

`engine::Project` and `Project::from_with_salsa_sync` are retained for
test/interpreter use only -- they are no longer on any production code path.

### Threading SimlinDb to Internal Callers

Functions that need simulation results gain a `db: &SimlinDb` (or `&dyn Db`)
parameter. Call sites affected:

- **`src/simlin-engine/src/layout/mod.rs`** -- auto-layout LTM loop scoring.
  With always-on LTM, the db already contains LTM compilation results.
- **`src/simlin-engine/src/analysis.rs`** -- `analyze_model()` for loop
  dominance.
- **`src/simlin-engine/src/project.rs`** LTM methods (`with_ltm`,
  `with_ltm_all_links`) -- eliminated entirely since LTM is always-on in the
  db.
- **`src/simlin-cli/src/main.rs`** -- CLI simulation creates a `SimlinDb`,
  syncs, and uses `compile_project_incremental`.

The AST-walking interpreter (`Simulation::run_to_end()`) is only used for
cross-validation in tests. It does not participate in production code paths.

### Deleting the Old Bytecode Compilation Path

With all callers migrated, the monolithic bytecode compilation path is removed:

- **Deleted:** `compile_project` (`src/simlin-engine/src/interpreter.rs:1625`),
  `Simulation::compile()` (`:1422`), `Module::compile()`
  (`src/simlin-engine/src/compiler/mod.rs:1227`), `build_metadata` (`:886`),
  `compile_simulation` (`src/libsimlin/src/lib.rs:555`).

- **Retained:** `Module::new` (`:1032`) for AST-walking interpreter,
  `Simulation::new` + `run_to_end` for cross-validation, `Opcode` enum and
  `ByteCode` in `src/simlin-engine/src/bytecode.rs` (VM still uses these),
  all `Expr` types and expression lowering (interpreter needs them).

## Existing Patterns

### Current Incremental Pipeline

The salsa-based incremental pipeline is fully operational for non-LTM
compilation. Per-variable tracked functions (`parse_source_variable`,
`variable_direct_dependencies`, `compile_var_fragment`) with symbolic bytecode
(`SymbolicOpcode`, `CompiledVarFragment`) and late assembly
(`assemble_module`, `assemble_simulation`). This design extends that pipeline
rather than introducing new patterns.

### LTM Tracked Functions

LTM analysis is already decomposed into tracked functions:
`model_causal_edges`, `model_loop_circuits`, `model_cycle_partitions`,
`link_score_equation_text`, `module_ilink_equation_text`, and higher-level
aggregators (`model_ltm_synthetic_variables`,
`model_ltm_all_link_synthetic_variables`, `module_ltm_synthetic_variables`).
These are tested in `src/simlin-engine/src/db_tests.rs` but not connected to
the compilation pipeline. This design connects them.

### Dual Error Paths

The codebase currently maintains both struct-field errors (on `engine::Project`,
`ModelStage1`, `Variable`) and the `CompilationDiagnostic` salsa accumulator,
with deduplication by `(model_name, variable_name, error_code)`. This design
eliminates the struct-field path, diverging from the current dual approach.

### BuiltinVisitor Module Expansion

`BuiltinVisitor::walk()` in `src/simlin-engine/src/builtins_visitor.rs`
recognizes stdlib module names and creates synthetic module variables. This
design removes `previous` and `init` from that path, converting them to
`BuiltinFn` variants instead. The pattern remains for SMOOTH, DELAY, and TREND.

## Additional Considerations

### LTM Array Support

LTM does not yet support models with arrayed variables. The `ltm_enabled` flag
on `SourceProject` allows disabling LTM for arrayed models. The north star is
always-on LTM with no measurable overhead; array support is a future extension
that this architecture accommodates (arrayed LTM vars would use the same
parallel compilation path with array-aware `compile_ltm_var_fragment`).

### Simulation Isolation

When `compile_project_incremental` produces a `CompiledSimulation`, the VM takes
a clone of the bytecode data. Subsequent edits to the db do not affect running
simulations. This guarantee is preserved -- identical to the current behavior.

### Determinism

Variable layouts use the same alphabetical ordering as the current
`build_metadata()`. LTM variables are sorted alphabetically within their
section. Loop IDs use the same deterministic assignment from
`assign_deterministic_loop_ids()`. Salsa memoization is transparent -- cached
results are identical to recomputed results.
