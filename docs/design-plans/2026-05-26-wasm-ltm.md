# Loops That Matter on the wasm Simulation Backend (wasm-ltm) Design

## Summary

LTM (Loops That Matter) analysis decomposes into a *numeric half* -- synthetic
`$⁚ltm⁚*` score equations the engine generates and folds into the `CompiledSimulation`,
so they run every timestep like any other variable -- and an *analytic half* of
post-simulation Rust that turns those score series into causal links, loop polarities,
relative loop scores, and (in discovery mode) discovered dominant loops. The bytecode VM
runs both halves over one in-memory `engine::Results`. The wasm backend lowers the same
bytecode opcode-for-opcode, so the synthetic equations should lower like any other; but
the wasm compile path never enables LTM today, the blob writes results into JS-owned
linear memory libsimlin never sees, and `@simlin/engine` rejects `enableLtm` on the wasm
engine outright (issue #636). This work removes those three blocks so that
`Model.simulate({ engine: 'wasm', enableLtm: true })` matches the VM with no silent
fallback.

The approach reuses existing seams rather than building new machinery. At *compile* time,
`compile_datamodel_to_artifact` and `simlin_model_compile_to_wasm` gain `ltm_enabled` /
`ltm_discovery_mode` flags that flip the same salsa setters `simlin_sim_new` already uses;
because the blob's `WasmLayout.var_offsets` is a verbatim copy of `CompiledSimulation.offsets`,
the `$⁚ltm⁚*` slots appear in the layout automatically once the synthetic vars exist. At
*run* time nothing changes -- the blob computes the score columns into its step-major slab
-- except that LTM's array/var multiplication can trip `MAX_UNROLL_UNITS` or an unlowerable
opcode, which surfaces as an explicit `WasmGenError::Unsupported` rather than a fallback.
At *analyze* time, the analytic half stays in Rust: the host hands the blob's slab plus the
serialized layout back across two new libsimlin functions that reconstruct an
`engine::Results` (mirroring `Vm::into_results()`), recompute the LTM structural snapshots
from salsa exactly as `simlin_sim_new` does, and drive the *existing* `ltm_post` /
`ltm_finding` / polarity code through a single backend-agnostic core shared with the VM
path. Funneling both backends through that one core is the design's structural defense
against the divergent-implementation bug class of #624 -- there is no TypeScript
reimplementation of the analysis; TS only marshals the slab and deserializes the result via
the existing `convertLinks`. The work is staged across six TDD phases, gated by a
`simulate_ltm`-style wasm-vs-VM parity harness whose floor of LTM-models-that-lower ratchets
upward as later phases broaden coverage, with heavy models `#[ignore]`d to respect the
3-minute `cargo test` cap.

## Definition of Done

Enable LTM (Loops That Matter) analysis on the wasm simulation backend so that
`@simlin/engine`'s `Model.simulate({ engine: 'wasm', enableLtm: true })` succeeds and
`getLinks()` / relative loop scores / discovery on a wasm sim match the VM within the
engine's existing tolerances -- with no silent VM fallback.

Deliverables:

1. **LTM threaded into the wasm compile.** `ltm_enabled` (and `ltm_discovery_mode`) flow
   through `compile_datamodel_to_artifact` and `simlin_model_compile_to_wasm`, set via
   `set_project_ltm_enabled`, so the blob's `WasmLayout` carries the `$⁚ltm⁚*` synthetic
   series automatically (the layout is a verbatim copy of `CompiledSimulation.offsets`).
2. **LTM wasm-parity harness (TDD).** A `simulate_ltm.rs` twin runs LTM corpus models
   through both backends under the DLR-FT interpreter with a monotonically rising floor;
   heavy models stay `#[ignore]`d to respect the 3-minute `cargo test` cap. It surfaces
   every `WasmGenError::Unsupported` and numeric divergence.
3. **Analytic half stays in Rust, fed by the blob's series.** A small, orthogonal
   libsimlin FFI ingests the blob's result slab + layout, reconstructs an `engine::Results`
   (with the LTM structural snapshots recomputed from salsa), and runs the *existing*
   `ltm_post` / `ltm_finding` / polarity code. No TypeScript reimplementation of the
   analysis (avoiding the divergent-implementation bug class of #624).
4. **`@simlin/engine` wiring (node + browser).** Drop the up-front `enableLtm` rejection
   and the `getLinks` wasm throw; populate `Run.links` for wasm; verified through both node
   `DirectBackend` and the browser Web Worker.
5. **No silent fallback.** An LTM model the wasm backend genuinely cannot lower returns an
   explicit `WasmGenError` (a clean error, never a panic or a silently-wrong result).

Scope decisions:

- **Discovery mode** (`ltm_discovery_mode`) is covered, isolated as its own implementation
  phase (it changes which synthetic vars the blob emits and requires running
  `discover_loops_with_graph` over the blob's link-score series).
- **Out of scope:** pysimlin / simlin-mcp wiring (the libsimlin FFI is shared, so adoption
  by other bindings is a later effort); changing the bytecode VM; generated-wasm
  performance / benchmarks.

## Acceptance Criteria

### wasm-ltm.AC1: LTM-enabled wasm compilation produces a blob carrying the LTM series
- **wasm-ltm.AC1.1 Success:** `compile_datamodel_to_artifact(..., ltm_enabled: true, ...)` on a scalar LTM model produces a `WasmLayout` whose `var_offsets` contains the `$⁚ltm⁚link_score⁚*` and `$⁚ltm⁚loop_score⁚*` entries.
- **wasm-ltm.AC1.2 Success:** the blob, run under the DLR-FT interpreter, writes the `$⁚ltm⁚*` columns; those series match the VM within the engine's existing tolerances.
- **wasm-ltm.AC1.3 Success:** `Model.simulate({ engine: 'wasm', enableLtm: true })` resolves to a `Sim` (the up-front rejection is gone) under node, and `getLinks()` returns links annotated with scores.
- **wasm-ltm.AC1.4 Success:** the same `engine:'wasm'` + `enableLtm` path through `WorkerBackend` (browser) yields `getLinks` scores matching node.
- **wasm-ltm.AC1.5 Edge:** compiling the same model with `ltm_enabled: false` produces a layout with no `$⁚ltm⁚*` entries (LTM-off behavior unchanged).

### wasm-ltm.AC2: Analytic outputs match the VM within tolerance
- **wasm-ltm.AC2.1 Success:** `simlin_analyze_links_from_wasm_results` returns per-link scores equal to `simlin_analyze_get_links` (VM) for a scalar LTM model, within tolerance.
- **wasm-ltm.AC2.2 Success:** `simlin_analyze_rel_loop_score_from_wasm_results` equals `simlin_analyze_get_relative_loop_score` (VM) for each loop id, including subscripted ids.
- **wasm-ltm.AC2.3 Success:** `getLinks()` on a wasm sim (node `DirectBackend`) returns scores matching `getLinks()` on a VM sim, and `Run.links` is populated for a wasm LTM run.
- **wasm-ltm.AC2.4 Success:** arrayed / cross-element LTM models that lower match the VM element-for-element (per-element link and loop scores).
- **wasm-ltm.AC2.5 Success:** in discovery mode, the discovered loops and their per-timestep scores (via `discover_loops_with_graph` over the blob-derived `Results`) match the VM's discovery output.

### wasm-ltm.AC3: No silent fallback for unlowerable LTM models
- **wasm-ltm.AC3.1 Failure:** an arrayed LTM model that exceeds `MAX_UNROLL_UNITS` (or uses an opcode the backend cannot lower) returns `WasmGenError::Unsupported` from the compile path — never a panic, never a silently-wrong blob.
- **wasm-ltm.AC3.2 Failure:** that unsupported case surfaces to the TS caller as a `SimlinError`/`Error` from `Model.simulate({ engine: 'wasm', enableLtm: true })`; the same model still simulates via `engine: 'vm'`.

### wasm-ltm.AC4: Parity harness with a ratcheting floor
- **wasm-ltm.AC4.1 Success:** the harness runs the LTM corpus through both backends, comparing wasm-vs-VM, and enforces a monotonically rising floor on the count of LTM models that lower; heavy models are `#[ignore]`d to respect the 3-minute cap.
- **wasm-ltm.AC4.2 Failure:** a regression that drops a previously-supported LTM model below the floor (or any `Unsupported` at the end-state gate) fails the suite.

### wasm-ltm.AC5: Engineering quality (cross-cutting)
- **wasm-ltm.AC5.1:** the VM and wasm analytic paths share one engine-level core; no analysis logic is reimplemented per-backend and none is reimplemented in TypeScript. The FFI grows by exactly the two `*_from_wasm_results` functions (no bulk/batch endpoint).
- **wasm-ltm.AC5.2:** new code reaches >=95% coverage via TDD; each new FFI function and each new lowering/feature group has unit tests.

## Glossary

### System dynamics / LTM
- **System dynamics (SD)**: A modeling discipline that simulates a system as stocks
  (accumulations), flows (rates), and feedback loops integrated over time; the domain
  Simlin's engine serves.
- **LTM (Loops That Matter)**: The engine's feedback-loop-scoring feature. It quantifies,
  per timestep, how much each causal link and feedback loop drives a model's behavior. It
  has a *numeric half* (synthetic score equations folded into the simulation) and an
  *analytic half* (post-run Rust that turns score series into links, polarities, and
  relative loop scores).
- **Causal link / loop polarity**: A causal link is a directed edge between two variables;
  its polarity (+/-) is whether an increase in the source increases or decreases the target.
  Loops are cycles of links; loop polarity (reinforcing vs. balancing) follows from the
  links' polarities.
- **Link score / loop score / relative loop score**: The LTM numeric outputs. A link score
  measures one link's per-timestep contribution; a loop score aggregates a loop's links; the
  relative loop score normalizes a loop's score against all loops at each timestep.
- **`$⁚ltm⁚*` synthetic series**: The hidden variables the engine generates for LTM --
  `$⁚ltm⁚link_score⁚{from}→{to}`, `$⁚ltm⁚loop_score⁚{id}`, `$⁚ltm⁚agg⁚{n}` -- that compile
  into the simulation as ordinary equations and run every timestep. (The `⁚` is the Unicode
  two-dot punctuation `\u{205A}`, used as an internal-name separator.)
- **Discovery mode (`ltm_discovery_mode`)**: An LTM mode that emits link-score series and
  then *discovers* the model's dominant feedback loops post-hoc (via
  `discover_loops_with_graph`) rather than scoring a pre-known loop set. It changes which
  synthetic vars the blob emits.
- **Loop structural snapshots (`loop_partitions` / `loop_element_index`)**: Salsa-derived
  structural metadata about which variables/elements belong to which loop, captured at
  compile time. The relative-loop-score math indexes the score series through these. The
  from-series FFI must recompute them exactly as `simlin_sim_new` does.
- **Arrayed / apply-to-all (A2A) / subscript / cross-element**: An arrayed variable is
  indexed over named dimensions; apply-to-all applies one equation element-wise, subscripting
  selects specific elements, and cross-element links connect distinct elements. Arrayed LTM
  multiplies the synthetic-var count per element, which is what stresses `MAX_UNROLL_UNITS`.

### This codebase
- **`simlin-engine`**: The core Rust crate that compiles, type-checks, and simulates SD
  models, producing one `CompiledSimulation` runnable by either backend.
- **Bytecode VM (`vm.rs`)**: The stack-based interpreter that executes compiled bytecode.
  It is the design's correctness oracle; the wasm path is parity-tested against it.
- **`wasmgen` backend**: The engine's code-generation path that lowers a `CompiledSimulation`
  into a self-contained, import-free WebAssembly module ("blob") that mirrors the VM
  opcode-for-opcode.
- **wasm blob / wasm engine**: The per-model WebAssembly module emitted by `wasmgen`,
  instantiated and driven by the host. Selected via `Model.simulate({ engine: 'wasm' })`.
- **`CompiledSimulation` / `CompiledSimulation.offsets`**: The engine's compiled, ready-to-run
  representation, consumed by both backends. `offsets` is the global canonical-variable-name
  -> slot map; the wasm layout copies it verbatim.
- **`WasmLayout` / `var_offsets`**: The blob's result-layout descriptor: geometry plus a
  name -> slot-offset map (`var_offsets`). Because `var_offsets` is a verbatim copy of
  `CompiledSimulation.offsets` with no name filter, the `$⁚ltm⁚*` slots appear automatically
  once they exist in the compiled sim.
- **Step-major slab**: The blob's results region in linear memory -- `n_chunks` rows of
  `n_slots` f64 values (time in column 0) -- so a variable's series is read at stride
  `n_slots`. The from-series FFI reconstructs a `Results` from this slab.
- **`engine::Results`**: The engine's in-memory result container (offsets + f64 slab). It is
  built from raw slabs in several non-VM places (`Vm::into_results()`, the VDF/CSV loaders);
  the from-series FFI rebuilds one from the wasm slab the same way.
- **salsa**: The incremental-computation framework the engine uses to cache compilation.
  `compile_project_incremental` is its production compile entry point; the LTM flags are set
  on the synced `SourceProject` before calling it.
- **`set_project_ltm_enabled` / `set_project_ltm_discovery_mode`**: The salsa setters that
  toggle LTM on the `SourceProject`. `simlin_sim_new` already does a temporary
  set-snapshot-reset dance with these; the wasm compile path follows the same pattern.
- **libsimlin**: The flat C FFI wrapper around `simlin-engine` (used by WASM, CGo, C/C++).
  The two new `*_from_wasm_results` functions and the LTM flags on
  `simlin_model_compile_to_wasm` live here.
- **`compile_datamodel_to_artifact` / `simlin_model_compile_to_wasm`**: The engine-level and
  FFI-level wasm compile entry points; both gain `ltm_enabled` / `ltm_discovery_mode` params.
- **`simlin_analyze_get_links` / `simlin_analyze_get_relative_loop_score`**: The existing
  VM-backed analyze FFI functions, refactored to build a shared core's inputs from their
  `SimlinSim`. The new `simlin_analyze_links_from_wasm_results` /
  `simlin_analyze_rel_loop_score_from_wasm_results` are their slab-fed twins.
- **`SimlinModel` / `SimlinSim`**: libsimlin's opaque handles. The from-series functions take
  a `SimlinModel` (structural/salsa half) plus the slab+layout (numeric half); they need no
  `SimlinSim`, so `SimlinSim` stays VM-only.
- **`ltm_post` / `ltm_finding`**: The Rust modules implementing the analytic half --
  relative-loop-score computation and loop discovery, respectively. They reach the score
  series through `results.offsets.get(...)` + striding, so a `Results` rebuilt from the blob
  drives them unchanged.
- **`Vm::into_results()`**: The VM's slab -> `engine::Results` conversion, mirrored by the
  from-series reconstruction.
- **`MAX_UNROLL_UNITS`**: The wasmgen per-function array-unrolling budget. LTM augmentation
  multiplies array/var counts, so an arrayed LTM model can exceed it and trip
  `WasmGenError::Unsupported`.
- **`WasmGenError::Unsupported`**: The clean error wasmgen returns for a construct it cannot
  lower -- never a panic, never a silently-wrong blob. Surfaces to the TS caller as a
  `SimlinError`/`Error`; no silent VM fallback.
- **Malloc-and-return-buffer convention**: libsimlin's pattern for returning variable-length
  data across the FFI (callee allocates, returns pointer+length, caller frees). The host uses
  it to copy the slab and serialized layout into libsimlin's linear memory.
- **`@simlin/engine`**: The TypeScript package exposing the WASM-compiled engine via a
  Promise-based API. Gains LTM-on-wasm support (drop the up-front `enableLtm` rejection and
  the `getLinks` wasm throw; populate `Run.links` for wasm).
- **`Sim` / `Model` / `Run` / `Run.links`**: The public `@simlin/engine` classes. `Run.links`
  holds a completed run's causal links; this work populates it for wasm LTM runs.
- **`DirectBackend` / `WorkerBackend` / `WorkerServer`**: The TS backends. `DirectBackend`
  (in-thread, used by node and inside the Worker) owns the blob's `WebAssembly.Instance` and
  layout and is where the wasm read/analyze path lives. `WorkerBackend` is the browser path;
  `WorkerServer` wraps a `DirectBackend`, so LTM-on-wasm needs no new worker message types.
- **`getLinks()`**: The `@simlin/engine` API returning a model's causal links annotated with
  LTM scores. Today it throws on the wasm engine; this work makes it read the blob's slab and
  call the from-series FFI.
- **`convertLinks` / `readStridedSeries`**: Existing TS helpers reused here -- `convertLinks`
  deserializes the FFI's link result; `readStridedSeries` reads a strided variable series out
  of the blob's linear memory.
- **`simulate_ltm.rs` / `simulate.rs` / parity harness / floor gate**: The Rust integration
  suites. The new `simulate_ltm`-style wasm twin runs LTM corpus models through both backends,
  diffs the `$⁚ltm⁚*` series and analytic outputs, and enforces a monotonically rising floor
  on the count of LTM models that lower (a regression below the floor fails the suite).
- **`analyze_model` (`analysis.rs`)**: The engine entry that runs LTM analysis end-to-end; it
  hard-codes the VM today, which is why discovery-mode parity (Phase 5) drives the wasm
  backend directly in a Rust test rather than through this entry.
- **C-LEARN / arms-race**: Large/heavy SD models in the corpus (e.g. a climate-policy model)
  whose wasm LTM/discovery parity tests are `#[ignore]`d to respect the test time cap.

### Third-party
- **WebAssembly (wasm) / linear memory**: A portable bytecode/VM target. The blob computes
  over a single flat f64 array (linear memory) the host reads results out of.
- **DLR-FT interpreter (`wasm-interpreter` + `checked`)**: A pure-Rust WebAssembly interpreter
  (from the DLR flight-software group), pinned by git rev, used in Rust tests to execute the
  emitted blob (validate -> instantiate via the `checked` `Store` -> invoke -> stride memory)
  and compare against the VM oracle. It is an interpreter, not a JIT, so heavy models run
  slowly -- hence `#[ignore]`.
- **`WebAssembly.Instance` / `Float64Array`**: A standard instantiated WebAssembly module
  (owned by `DirectBackend`) and the typed array the host reads the strided results into.

## Architecture

LTM has two halves. The **numeric half** is a set of synthetic equations
(`$⁚ltm⁚link_score⁚{from}→{to}`, `$⁚ltm⁚loop_score⁚{id}`, `$⁚ltm⁚agg⁚{n}`) the
engine generates and compiles into the simulation; they are "just more equations" that
run every timestep. The **analytic half** is post-simulation Rust that turns those score
series into causal links, loop polarities, relative loop scores, and (in discovery mode)
discovered dominant loops. On the bytecode VM both halves share one in-memory
`engine::Results`. The wasm backend lowers the same `CompiledSimulation` bytecode
opcode-for-opcode, so the synthetic equations should lower like any other; but the blob
writes results into JS-owned linear memory libsimlin never sees, and today the wasm
compile path never enables LTM at all.

The design has three seams, matching the three gaps in issue #636:

1. **Compile** — flip `ltm_enabled` (and `ltm_discovery_mode`) on the synced
   `SourceProject` before `compile_project_incremental`, so the blob's bytecode *and* its
   `WasmLayout.var_offsets` carry the `$⁚ltm⁚*` slots. The layout is a verbatim copy of
   `CompiledSimulation.offsets` (`wasmgen/module.rs:824`), so once the synthetic vars exist
   in the compiled sim they appear in the layout automatically — no layout-format change.

2. **Run** — unchanged. The blob computes the `$⁚ltm⁚*` columns into its step-major
   slab alongside every other variable. The only new failure mode is `WasmGenError::
   Unsupported`: LTM augmentation multiplies array/var counts, so an arrayed LTM model can
   trip `MAX_UNROLL_UNITS` (`wasmgen/lower.rs:1106`) or an unlowerable opcode. Per the
   engine-wasm-sim contract there is **no** silent VM fallback — the error propagates as a
   `SimlinError` to the caller.

3. **Analyze** — the host hands the blob's slab + serialized layout back across the FFI.
   Two new functions reconstruct an `engine::Results` from that slab (mirroring
   `Vm::into_results()`, `vm.rs:897`), recompute the LTM structural snapshots
   (`loop_partitions` / `loop_element_index`) from salsa exactly as `simlin_sim_new` does
   (`libsimlin/src/simulation.rs:83-132`), and call a shared, backend-agnostic engine-level
   analysis core. Because the VM path and the wasm path funnel through the *same* core, the
   analysis cannot diverge between backends — the explicit anti-goal carried over from #624.

**FFI contract (new functions in `libsimlin/src/analysis.rs`):**

```c
// Per-link causal structure (edges + polarity from salsa) annotated with the
// $⁚ltm⁚link_score series read out of the supplied wasm slab. Mirrors the
// return type of the existing simlin_analyze_get_links, so TS reuses convertLinks.
SimlinLinks *simlin_analyze_links_from_wasm_results(
    SimlinModel *model,
    const uint8_t *slab_ptr, size_t slab_len,        // blob's step-major f64 results region
    const uint8_t *layout_ptr, size_t layout_len,    // serialized WasmLayout (names -> slots)
    SimlinError **out_error);

// Relative loop score series for one (possibly subscripted) loop id, derived
// post-hoc from the $⁚ltm⁚loop_score series in the supplied slab. Mirrors the
// out-parameter shape of the existing simlin_analyze_get_relative_loop_score.
void simlin_analyze_rel_loop_score_from_wasm_results(
    SimlinModel *model,
    const uint8_t *slab_ptr, size_t slab_len,
    const uint8_t *layout_ptr, size_t layout_len,
    const char *loop_id,
    double *results_ptr, size_t len, size_t *out_written,
    SimlinError **out_error);
```

Both take a `SimlinModel` (for the structural/salsa half) plus the slab+layout (for the
numeric series); neither needs a `SimlinSim`, so `SimlinSim` stays VM-only. The existing
`simlin_analyze_get_links` / `simlin_analyze_get_relative_loop_score` are refactored to
build the same core's inputs from their `SimlinSim`, so both backends share one
implementation.

**Compile-path contract (threading the LTM flags):**

```rust
// src/simlin-engine/src/wasmgen/module.rs
pub fn compile_datamodel_to_artifact(
    datamodel: &crate::datamodel::Project,
    model_name: &str,
    ltm_enabled: bool,          // new
    ltm_discovery_mode: bool,   // new
) -> Result<WasmArtifact, WasmGenError>;
```

```c
// src/libsimlin/src/model.rs -- two new flag params; outputs/return unchanged
void simlin_model_compile_to_wasm(
    SimlinModel *model,
    bool ltm_enabled,           // new
    bool ltm_discovery_mode,    // new
    uint8_t **out_wasm, size_t *out_wasm_len,
    uint8_t **out_layout, size_t *out_layout_len,
    SimlinError **out_error);
```

The TS wrapper `simlin_model_compile_to_wasm` (`engine/src/internal/wasmgen.ts:92`) gains
the two flags; `DirectBackend.simNewWasm` forwards the sim's `enableLtm` (discovery is not
TS-exposed, so it passes `false` from TS).

**Host read/analyze path (TypeScript, in `DirectBackend`).** When `getLinks()` is called
on a wasm LTM sim: read the blob's results region from its `WebAssembly.Instance` memory
into a `Float64Array`, copy that slab and the serialized layout into libsimlin's linear
memory (malloc convention), call `simlin_analyze_links_from_wasm_results`, deserialize via
the existing `convertLinks`, and free the buffers. Analysis is on-demand (not the per-frame
scrub path), so copying the slab back is acceptable, as the issue notes.

## Existing Patterns

Investigation grounded every touch point in current code; this design slots into existing
seams rather than introducing new machinery.

- **Threading `ltm_enabled` through the salsa compile.** `set_project_ltm_enabled` /
  `set_project_ltm_discovery_mode` (`db.rs:5552`/`5563`) are the established setters; the
  compile pipeline already reads the flags at three gated points (`db.rs:2542` layout,
  `db.rs:4469` runlist, `db_ltm.rs` generation). `simlin_sim_new` already does the
  temporary set-snapshot-reset dance (`simulation.rs:64-138`) — the wasm compile path and
  the from-series FFI follow the same pattern.
- **`WasmLayout` is already LTM-agnostic.** `var_offsets` is a verbatim copy of
  `sim.offsets` (`wasmgen/module.rs:824`); no filter, no name predicate. LTM slots appear
  automatically once present in the compiled sim.
- **`engine::Results` is built from raw slabs in 5+ non-VM places** — `Vm::into_results()`
  (`vm.rs:897`), the VDF loader (`vdf.rs:1995`), `.dat`/CSV loaders (`compat.rs:185`/`243`),
  and LTM test helpers. All fields are `pub` and the type is re-exported (`lib.rs:133`).
  Reconstructing one from the wasm slab follows these.
- **The analytic half is mostly backend-independent already.** Causal edges
  (`model_causal_edges`), link polarity (`compute_link_polarities`,
  `db_analysis.rs:1627`), and structurally-detected loops (`model_detected_loops`,
  `db_analysis.rs:1556`) are pure salsa queries on the model handle — `simlin_analyze_
  get_loops` and the structural half of `simlin_analyze_get_links` need no change. Only the
  score series come from `Results`, and every consumer (`ltm_post::*`, `ltm_finding::*`)
  reaches them through just `results.offsets.get(&<canonical ltm name>)` + step-major
  striding, so a `Results` rebuilt from the blob drives them unchanged.
- **DLR-FT interpreter parity harness.** `libsimlin/tests/wasm.rs` is the template:
  compile to wasm, `validate` → instantiate via the `checked` `Store` → invoke `run` →
  stride the results region, compare to the VM oracle. The LTM parity twin reuses this
  stride loop to build a `Results` from the blob.
- **`simulate_ltm.rs` structure.** Its `compile_ltm_incremental_with_partitions`
  (`tests/simulate_ltm.rs:34`) and `ensure_ltm_results` (via
  `ltm_post::compute_rel_loop_scores`, `:138`) are the comparison machinery the wasm twin
  reuses, swapping the `Vm::new(...).into_results()` block for "run the blob, stride into a
  `Results`."
- **TS backend demux.** `DirectBackend` (`engine/src/direct-backend.ts`) already branches
  every sim op on the handle's `engine` kind and owns the wasm `WebAssembly.Instance` +
  layout; `convertLinks` (`:131`) + `readStridedSeries` (`internal/wasmgen.ts:207`) are the
  existing read paths reused here.

**Divergence:** none. The wasmgen backend's `CLAUDE.md` currently lists "LTM (VM-only)" as
out of scope; this design removes that limitation and that doc line is updated in the final
phase.

## Implementation Phases

Six phases, TDD throughout (95%+ coverage on new code). The LTM parity harness is introduced
in Phase 1 and ratchets its floor upward as later phases broaden coverage; heavy models stay
`#[ignore]`d to respect the 3-minute `cargo test` cap.

<!-- START_PHASE_1 -->
### Phase 1: Thread LTM into the wasm compile + numeric-series parity harness
**Goal:** A wasm blob compiled with LTM enabled carries the `$⁚ltm⁚*` series in its layout,
and a both-backends harness proves the synthetic equations lower and match the VM numerically
for scalar LTM models.

**Components:**
- `src/simlin-engine/src/wasmgen/module.rs` — `compile_datamodel_to_artifact` gains
  `ltm_enabled` / `ltm_discovery_mode` params; calls `set_project_ltm_enabled` /
  `set_project_ltm_discovery_mode` before `compile_project_incremental`. The thin
  `compile_datamodel_to_wasm` wrapper updated accordingly.
- `src/libsimlin/src/model.rs` — `simlin_model_compile_to_wasm` gains the two flag params
  (outputs/return unchanged).
- `src/simlin-engine/tests/` — a `simulate_ltm`-style parity harness that, for each LTM
  corpus model, compiles with LTM enabled, runs the VM and the blob (DLR-FT interpreter),
  reconstructs a `Results` from the blob slab, and diffs the `$⁚ltm⁚*` series; maintains a
  monotonically rising floor of LTM-models-that-lower. Surfaces every `WasmGenError::
  Unsupported` (skip-not-fail during rollout).

**Dependencies:** None.

**Done when:** scalar LTM corpus models lower to wasm and their `$⁚ltm⁚*` series match the
VM within existing tolerances; the floor gate is active; `cargo test` passes. Covers
`wasm-ltm.AC1.1`, `wasm-ltm.AC1.2`, `wasm-ltm.AC4.1`.
<!-- END_PHASE_1 -->

<!-- START_PHASE_2 -->
### Phase 2: Backend-agnostic analytic core + from-series FFI
**Goal:** The existing analysis runs on a `Results` reconstructed from a wasm slab, behind a
small from-series FFI, sharing one implementation with the VM path.

**Components:**
- `src/libsimlin/src/analysis.rs` — extract the post-acquisition bodies of
  `simlin_analyze_get_links` and `simlin_analyze_get_relative_loop_score` into a shared core
  taking `(db, model, source_project, results: &Results, snapshots, enable_ltm)`; the
  existing VM functions build the core's inputs from their `SimlinSim`.
- `src/libsimlin/src/analysis.rs` — `simlin_analyze_links_from_wasm_results` and
  `simlin_analyze_rel_loop_score_from_wasm_results` (signatures in Architecture): rebuild
  `Results` from slab+layout (canonicalize via `engine::canonicalize`), recompute
  `loop_partitions` / `loop_element_index` from salsa (same calls as `simlin_sim_new`), call
  the shared core.
- A shared `Results`-from-`WasmLayout`+slab helper (reused by the harness and the FFI).
- `src/libsimlin/tests/wasm.rs` — FFI parity twin: compile an LTM model to wasm, run the
  blob, call the from-series FFI, compare links + scores against the VM analyze FFI.

**Dependencies:** Phase 1.

**Done when:** the from-series FFI produces links/scores and relative loop scores matching
the VM analyze FFI for scalar LTM models; no logic is duplicated between backends; tests
pass. Covers `wasm-ltm.AC2.1`, `wasm-ltm.AC2.2`, `wasm-ltm.AC5.1`.
<!-- END_PHASE_2 -->

<!-- START_PHASE_3 -->
### Phase 3: @simlin/engine node wiring
**Goal:** `Model.simulate({ engine: 'wasm', enableLtm: true })` succeeds under node and
`getLinks()` returns VM-matching scores; `Run.links` is populated for wasm.

**Components:**
- `src/engine/src/internal/wasmgen.ts` + `src/engine/src/direct-backend.ts` — pass
  `enableLtm` through `simlin_model_compile_to_wasm`; drop the up-front `enableLtm` rejection
  in `simNewWasm` (`direct-backend.ts:463`).
- `src/engine/src/direct-backend.ts` — replace the `getLinks` wasm throw (`:670`) with the
  wasm read/analyze path: stride the blob's slab, copy slab+layout into libsimlin, call
  `simlin_analyze_links_from_wasm_results`, reuse `convertLinks`, free buffers.
- `src/engine/src/sim.ts` — drop the `_engine !== 'wasm'` guard in `getRun` (`:234`) so a
  wasm LTM run populates `Run.links`.
- `src/engine/tests/` — jest parity tests: VM vs wasm `getLinks` scores; `Run.links`
  populated; the unsupported-model error case.

**Dependencies:** Phase 2.

**Done when:** a node `DirectBackend` runs LTM models through both engines with matching
`getLinks` scores and populated `Run.links`; tests pass. Covers `wasm-ltm.AC1.3`,
`wasm-ltm.AC2.3`.
<!-- END_PHASE_3 -->

<!-- START_PHASE_4 -->
### Phase 4: Arrayed/cross-element LTM + Unsupported hardening
**Goal:** Arrayed (A2A / subscripted / cross-element) LTM models either match the VM or fail
with a clean explicit error, and the floor ratchets up to include the supported ones.

**Components:**
- `src/simlin-engine/tests/` — extend the parity harness with arrayed corpus models
  (`test/arrayed_population_ltm/`, `test/cross_element_ltm/`): per-element link/loop score
  parity where they lower; assert a clean `WasmGenError::Unsupported` (no panic, no wrong
  result) where they trip `MAX_UNROLL_UNITS` or an unlowerable opcode.
- `src/engine/tests/` — assert the unsupported case surfaces as a `SimlinError` to the TS
  caller (no silent VM fallback).
- Ratchet the floor to include the now-supported arrayed LTM models.

**Dependencies:** Phases 2-3.

**Done when:** supported arrayed LTM models match the VM element-for-element through both the
series harness and the from-series FFI; genuinely-unlowerable ones return an explicit error
end-to-end; the floor includes them. Covers `wasm-ltm.AC2.4`, `wasm-ltm.AC3.1`,
`wasm-ltm.AC3.2`, `wasm-ltm.AC4.2`.
<!-- END_PHASE_4 -->

<!-- START_PHASE_5 -->
### Phase 5: Discovery mode (Rust-internal parity)
**Goal:** Discovery-mode loop discovery over the wasm blob's link-score series matches the
VM.

**Components:**
- `src/simlin-engine/tests/` — a discovery parity test: compile a discovery-mode LTM model
  to wasm (the `ltm_discovery_mode` flag threaded in Phase 1), run the blob, reconstruct a
  `Results`, feed it to `discover_loops_with_graph` (`ltm_finding.rs:710`), and compare the
  discovered loops + per-timestep scores against the VM's discovery output. The test drives
  the wasm backend directly because `analyze_model` (`analysis.rs:137`) hard-codes the VM.
- Confirm the discovery-mode synthetic vars (link scores) lower; `#[ignore]` the heavy
  discovery models (e.g. C-LEARN, arms-race) for the time cap.

**Dependencies:** Phases 1-2 (compile flag + `Results` reconstruction).

**Done when:** discovered loops and their scores match the VM for the discovery corpus
(heavy ones `#[ignore]`d); no FFI/TS surface added. Covers `wasm-ltm.AC2.5`.
<!-- END_PHASE_5 -->

<!-- START_PHASE_6 -->
### Phase 6: Browser/worker verification + docs
**Goal:** LTM-on-wasm works through the Web Worker with no protocol change, and the docs
reflect that wasmgen now supports LTM.

**Components:**
- `src/engine/tests/` — a `WorkerBackend` parity test for `engine:'wasm'` + `enableLtm`:
  `getLinks` over the worker matches node (and the VM). No new message types — `getLinks`
  already traverses the worker for the VM, and `DirectBackend` runs inside `WorkerServer`.
- `src/simlin-engine/src/wasmgen/` `CLAUDE.md` and the two wasm design plans — update the
  "LTM (VM-only)" / out-of-scope notes to reflect support.
- `docs/` index updates if needed.

**Dependencies:** Phases 3-4.

**Done when:** a `WorkerBackend` LTM-on-wasm `getLinks` matches node; no protocol change;
docs updated. Covers `wasm-ltm.AC1.4`.
<!-- END_PHASE_6 -->

## Additional Considerations

**No silent fallback (carried from engine-wasm-sim).** Every unlowerable LTM construct must
surface as an explicit `WasmGenError::Unsupported` → `SimlinError` → TS `Error`, never a
panic or a silently-wrong result. The harness deliberately includes models expected to trip
`MAX_UNROLL_UNITS` to prove the clean-error path.

**Anti-divergence is structural, not incidental.** The single shared analytic core (Phase 2)
is the mechanism that prevents the VM and wasm analyses from drifting apart — the exact
failure mode of #624. The design forbids reimplementing any analysis in TypeScript; TS only
marshals the slab and deserializes the result.

**Analysis requires a completed run.** The from-series functions assume the slab holds a
full run (`step_count = n_chunks`), matching how `getLinks` is called after `runToEnd`. They
do not support analyzing a partially-advanced (`run_to`) blob; that is out of scope.

**Relative loop scores and discovery are FFI/Rust-internal.** Neither is wired into the
public `@simlin/engine` API today (the `simlin_analyze_get_relative_loop_score` TS binding
is dormant; discovery is reachable only via `analyze_model` → `simlin-mcp-core`). Their
parity is therefore validated at the FFI/Rust harness level. The rel-loop-score from-series
function is added for symmetry and future consumers (e.g. pysimlin), not because a TS caller
needs it yet.

**Out of scope.** pysimlin / simlin-mcp wiring (the libsimlin FFI is shared, so adoption by
other bindings is a later effort); changing the bytecode VM; generated-wasm performance.
