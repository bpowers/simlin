# Loops That Matter on the wasm Simulation Backend (wasm-ltm) ŌĆö Human Test Plan

This plan covers manual verification of LTM-on-wasm: `Model.simulate({ engine:
'wasm', enableLtm: true })` and `Model.run(..., { engine: 'wasm', analyzeLtm:
true })` end-to-end, including `sim.getLinks()` / `Run.links`, the
`WorkerBackend` round-trip, and the no-silent-fallback contract on an
unsupported LTM model. All 16 acceptance criteria (`wasm-ltm.AC1.1` ..
`wasm-ltm.AC5.2`) already have automated coverage (see
`docs/implementation-plans/2026-05-26-wasm-ltm/test-requirements.md`); this plan
exists to (a) re-run the automated gates as a release checklist, (b) drive the
heavy `#[ignore]`d Rust twins out of the default suite, and (c) exercise the
user-visible behavior a human can judge from a Node REPL ŌĆö that LTM analysis on
`engine: 'wasm'` produces VM-faithful link sets and per-step scores, that the
WorkerBackend (browser-shaped) path agrees byte-for-byte with the node
DirectBackend, and that an unsupported LTM model surfaces a thrown
`SimlinError` instead of silently falling back to the VM.

The bytecode VM is the correctness oracle throughout; the wasm-LTM path is
held to VM parity within `1e-6` element-for-element on the `$ŌüÜltmŌüÜ*` columns
(the analytic half is identical Rust code shared by both backends).

## Prerequisites

- Run `./scripts/dev-init.sh` from the repo root (idempotent).
- A current WASM build of libsimlin so the engine tests can load
  `src/engine/core/libsimlin.wasm`: `pnpm build` (or the WASM build step).
- These default suites pass (the regression baseline):
  - Rust engine-side parity + Unsupported + floor gate:
    `cargo test -p simlin-engine --features file_io --test simulate_ltm_wasm`
  - libsimlin FFI parity: `cargo test -p simlin --test wasm`
  - TS engine LTM-on-wasm suite (node + worker):
    `pnpm -C src/engine test -- --testPathPattern='wasm-ltm|worker-wasm'`
- Model files present (used below):
  - `test/logistic_growth_ltm/logistic_growth.stmx` (scalar LTM, feedback loop)
  - `test/arms_race_3party/arms_race.stmx` (scalar LTM, used for discovery
    parity)
  - `test/decoupled_stocks/decoupled.stmx` (scalar LTM)
  - `test/arrayed_population_ltm/arrayed_population.stmx` (A2A, N=3)
  - `test/cross_element_ltm/cross_element.stmx` (cross-element, N=2)
  - `test/ltm_dynamic_range_unsupported/model.stmx` (LTM-on-wasm Unsupported
    fixture: feedback loop + `SUM(source[lo:hi])`)
  - `test/xmutil_test_models/C-LEARN v77 for Vensim.mdl` and
    `test/metasd/WRLD3-03/wrld3-03.mdl` (heavy ignored twins)

## End-to-End: LTM on the wasm engine through the TypeScript API

Purpose: validate the user-visible LTM-on-wasm surface end-to-end ŌĆö instantiate
a sim with `engine: 'wasm'` and `enableLtm: true`, read `sim.getLinks()`, and
confirm the scored link set matches the VM oracle for a small feedback model.
This spans the engine selection ABI (AC1.3), the analytic FFI (AC2.1/AC2.3),
and the Run-level facade (`Run.links`).

Setup: a tiny Node script (or a Node REPL) against the built `@simlin/engine`,
configured with `src/engine/core/libsimlin.wasm`, opening
`test/logistic_growth_ltm/logistic_growth.stmx`.

Steps and expected results:

1. Open the project, get the main model. Create two sims for the same model,
   both with LTM enabled:
   `vm = await model.simulate({}, { engine: 'vm', enableLtm: true })` and
   `wasm = await model.simulate({}, { engine: 'wasm', enableLtm: true })`.
   - Expected: both resolve to `Sim` instances with `sim.ltmEnabled === true`;
     no throw. AC1.3 (no up-front rejection on the wasm path) and AC5.1
     (single analytic core).
2. `await wasm.runToEnd()` and `await vm.runToEnd()`. Then call
   `await wasm.getLinks()` and `await vm.getLinks()`. Compare:
   - Same `(from, to)` set (any order).
   - Identical polarities per edge.
   - Each `link.score` is a `Float64Array` of length `await wasm.getStepCount()`,
     and `Math.abs(wasm.score[i] - vm.score[i]) <= 1e-6` for every step.
   - At least one link has a defined `score` (the feedback loop has a non-self
     edge that genuinely scores). AC2.1 / AC2.3.
3. From the same wasm `Sim`, call `await wasm.getStepCount()` and read one
   variable series: `await wasm.getSeries('population')`. Expected: a non-empty
   `Float64Array` whose length equals the step count, matching the VM's
   `population` series within `1e-9` (the eval-parity tolerance, tighter than
   the LTM tolerance because this is bytecode parity, not analysis parity).
4. Now exercise the `Run.links` facade (analyzeLtm = true on `model.run`):
   `wasmRun = await model.run({}, { analyzeLtm: true, engine: 'wasm' })` and
   `vmRun = await model.run({}, { analyzeLtm: true, engine: 'vm' })`. Expected:
   `wasmRun.links.length > 0`; the `(from, to)` set and polarities match
   between `wasmRun.links` and `vmRun.links`; every per-link `score` agrees
   within `1e-6`. AC2.3.
5. Dispose all sims (`await sim.dispose()`) and the project. Expected: no
   error, no resource warnings.

## End-to-End: No silent VM fallback on an unsupported LTM model

Purpose: validate the AC3.1 / AC3.2 contract ŌĆö that an LTM model the wasm
backend cannot lower surfaces as a thrown `Error`/`SimlinError` from
`Model.simulate({ engine: 'wasm', enableLtm: true })`, NOT a silent fall-back to
the VM. The same model must still simulate cleanly on `engine: 'vm'`.

Setup: the same Node script, this time opening
`test/ltm_dynamic_range_unsupported/model.stmx`. This fixture combines a
one-stock feedback loop (so LTM is genuinely enabled and the VM emits link/loop
scores) with `SUM(source[lo:hi])`, a non-constant subscript range
(`ViewRangeDynamic`, GH #612) the fully-unrolled wasm emitter cannot express.

Steps and expected results:

1. `await model.simulate({}, { engine: 'wasm', enableLtm: true })`.
   - Expected: the promise rejects with an `Error` (or `SimlinError`) whose
     message describes the codegen failure (mentions `ViewRangeDynamic` or
     "code generation failed"). The exact text is allowed to evolve; the only
     load-bearing assertion is that the call rejects, never silently resolves
     to a VM-backed sim.
2. `vmSim = await model.simulate({}, { engine: 'vm', enableLtm: true })`,
   then `await vmSim.runToEnd()`.
   - Expected: resolves cleanly; `await vmSim.getStepCount() > 0`;
     `(await vmSim.getLinks()).length > 0`. This confirms the limitation is
     wasm-backend-specific (the model itself is fine), and that there was no
     silent fallback at step 1 (otherwise step 1 would have succeeded the same
     way).

## End-to-End: LTM on the wasm engine through the WorkerBackend (browser-shaped)

Purpose: validate the AC1.4 promise ŌĆö `engine: 'wasm'` + `enableLtm: true`
survives the worker boundary. The `WorkerBackend` is the path the browser app
uses; the test harness wires a real `WorkerBackend` to a real `WorkerServer`
(which wraps a `DirectBackend`) through an in-memory loopback, so the
postMessage protocol, the FIFO queue, and `handleResponse` / `deserializeError`
are all exercised on real code paths.

Setup: a Node script using `WorkerBackend` + `WorkerServer` via
`createWorkerWasmPair()` (the helper in `src/engine/tests/worker-wasm.test.ts`).
Load `test/logistic_growth_ltm/logistic_growth.stmx`.

Steps and expected results:

1. Through the WorkerBackend, call
   `simHandle = await backend.simNew(modelHandle, true /* enableLtm */, 'wasm')`
   then `await backend.simRunToEnd(simHandle)`.
   - Expected: both calls resolve; no protocol error, no rejection.
2. `workerLinks = await backend.simGetLinks(simHandle)`. Compare against a
   sibling `DirectBackend` driving the same wasm blob on the same model with
   `enableLtm: true`.
   - Expected: identical `(from, to)` set; identical polarities;
     `Array.from(workerLink.score)` exactly equal to
     `Array.from(directWasmLink.score)` (same compiled blob and same analytic
     core ŌćÆ byte-identical, not just within `1e-6`).
3. Compare against a sibling VM run.
   - Expected: same `(from, to)` set and polarities; scores agree within `1e-6`
     (VM-vs-wasm tolerance).
4. Inspect the worker requests captured by the loopback: confirm that the
   served `simNew` request carries `engine: 'wasm'` and `enableLtm: true`,
   and that no new message-type strings appeared on the wire (the protocol
   delta is purely additive). AC1.4 / AC5.1.

## End-to-End: Arrayed and cross-element LTM through the wasm engine

Purpose: validate that the arrayed/cross-element fixtures (the corpus the
Rust harness covers in `series_arrayed_population_matches_vm` and
`series_cross_element_matches_vm`) also work through the TypeScript API and
produce VM-faithful per-element link/loop scores. AC2.4 is already pinned by
the Rust whole-slab comparator within `1e-6`; this is a smoke test that the
same parity surface holds at the TS public API. Skip if you already trust the
Rust comparator ŌĆö it is the authoritative judge for AC2.4.

Setup: the same Node script, opening
`test/arrayed_population_ltm/arrayed_population.stmx` (A2A, N=3) and
`test/cross_element_ltm/cross_element.stmx` (cross-element, N=2).

Steps and expected results (per fixture):

1. `wasm = await model.simulate({}, { engine: 'wasm', enableLtm: true })`,
   `vm = await model.simulate({}, { engine: 'vm', enableLtm: true })`,
   `await wasm.runToEnd()`, `await vm.runToEnd()`.
   - Expected: both resolve and complete.
2. `wasmLinks = await wasm.getLinks()`, `vmLinks = await vm.getLinks()`.
   - Expected: same `(from, to)` set (the cross-element fixture's per-element
     edges visit individual slots, so expect subscripted endpoints like
     `population[NYC]` to appear); same polarities; every per-link `score`
     agrees within `1e-6`.
